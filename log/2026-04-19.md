## Today's Work

### Split-Horizon DNS — Investigated, Deferred

Investigated long-term split-horizon DNS for `argocd.cloudhaven.work` → internal Tailscale IP (`100.78.197.38`).

**Approach evaluated:** Tailscale `extraRecords` in the ACL `dnsConfig` block — static A records served directly by MagicDNS (100.100.100.100) to tailnet devices. No separate DNS server needed.

**Blocker:** `tailscale_acl` resource at provider `~> 0.17` validates ACL JSON against a strict internal schema that doesn't include `dnsConfig`. Bumping to `~> 0.18` broke `tailscale_oauth_client` (resource removed or renamed in that version). Reverted both changes.

**Decision:** Deferred. `argocd.cloudhaven.work` resolves correctly over Tailscale without any host file changes — no split-horizon DNS needed right now. Revisit when there are multiple internal services (Grafana, etc.) — at that point do a proper provider version audit across all tailscale roots simultaneously.

**Note on architecture:** External services (landing page, QA app, OIDC) stay on `humboldt-external` → Cloudflare tunnel → public DNS. Only admin tools go internal. When split-horizon is revisited, CoreDNS is NOT needed — `extraRecords` in the ACL is the right approach.

### ArgoCD — nebulosa Cluster Registration — Completed

End-to-end path now works. ArgoCD on humboldt has `nebulosa` registered as a downstream cluster.

**Modules built:**
- `modules/argocd/token/` — creates ServiceAccount `argocd-manager` + `ClusterRoleBinding` + long-lived `kubernetes.io/service-account-token` Secret on the target cluster, writes bearer token to SSM `/restricted/argocd/{cluster_name}/token`.
- `modules/argocd/bootstrapper/` — reads the token from SSM, creates an `argocd_cluster` resource per entry in the `clusters` map.

**Wired in `platform/humboldt/argocd/`:**
- `main.tf`: calls `nebulosa_token` module (kubernetes.nebulosa provider) and `bootstrapper` (argocd provider).
- `providers.tf`: argocd provider on `argocd.cloudhaven.work:80` with `plain_text = true`, `grpc_web = true`, `auth_token = var.argocd_token`.

**Blockers hit, in order:**
1. `argocd-cm` missing `apiKey` capability for the `admin` account — patched `accounts.admin: apiKey, login` so tokens can be generated.
2. `TF_VAR_argocd_token` not set → provider hung indefinitely making gRPC call with empty token (no connection error).
3. gRPC provider hung even with token set — Envoy HTTP listener on port 80 does not properly pass h2c prior-knowledge. **Fix: `grpc_web = true` on the provider.** gRPC-web rides on HTTP/1.1 with `content-type: application/grpc-web+proto`; ArgoCD server handles it natively on the same port.
4. `argocd_cluster` apply errored with `specifying a root certificates file with the insecure flag is not allowed` — rejected the combo. Dropped `insecure` entirely; went for a correct-cert path instead.
5. `unable to parse bytes as PEM block` — SSM stores base64-encoded PEM; argocd provider's `ca_data` attribute expects raw PEM. Added `base64decode(...)` in the caller.
6. `dial tcp 10.244.3.27:6443: i/o timeout` — ArgoCD pods correctly resolved the egress service DNS to the Tailscale proxy pod, but the proxy pod itself couldn't reach `100.86.215.90`. Root cause: the operator's OAuth client minted proxy devices with `tag:k8s-humboldt`, and the ACL had no rule for that tag. Renamed the operator's device tag to `tag:k8s-worker` so the existing `tag:k8s-worker → tag:k8s-cp:*` ACL rule covers all operator-managed proxies. Dropped unused `tag:k8s-humboldt` from `tagOwners`. Re-tagged the existing proxy device via the Tailscale admin console (no pod/secret deletion needed).
7. cluster server URL used the egress service DNS name, which wasn't in the k3s apiserver cert SANs. Added `tls-san: [nebulosa-apiserver.argocd.svc.cluster.local]` to `/etc/rancher/k3s/config.yaml` on pi-1, restarted k3s — apiserver leaf cert regenerated under the same CA (so `ca_data` in SSM stayed valid).

**Final apply:** `Creation complete after 2s`. Cluster visible in ArgoCD UI (`Settings → Clusters`): `nebulosa` + `in-cluster`. Connection status shows `Unknown` until an app targets the cluster — that's expected; ArgoCD doesn't proactively probe.

**Rules learned:**
- For ArgoCD Terraform provider over HTTP gateway: use `grpc_web = true`. h2c does not survive Envoy's default HTTP listener, and it definitely doesn't survive Cloudflare tunnels.
- To reach a tailnet-only IP from in-cluster workloads, create a `Service` with annotation `tailscale.com/tailnet-ip`; operator creates a proxy pod (its own tailnet device) and rewrites the Service to ExternalName → proxy. The proxy device is subject to your normal ACL rules, so the proxy's tag must be granted access to the destination tag.
- When changing device tags in Tailscale, edit the device directly in the admin console — it's retroactive. Don't bother with pod/secret deletion to force re-registration.
- k3s apiserver cert SANs are configured via `tls-san:` in `/etc/rancher/k3s/config.yaml`. Adding an entry and restarting k3s regenerates the leaf cert under the same CA — existing CA data in downstream configs stays valid.

### Priority 1 — Talos EBS CSI (unblocks Phase 2 observability)

Install `aws-ebs-csi-driver` on the Talos/EC2 cluster (`humboldt` — this is the AWS-side cluster). This is a pure-Kubernetes install; Talos' immutability is fine because the driver is pods, not host daemons.

**Design notes (do all of these, not just install):**
- **IRSA:** create a role with the AWS-managed `AmazonEBSCSIDriverPolicy`; annotate the controller ServiceAccount. Use the existing IRSA module (`modules/aws/iam/app_role`).
- **StorageClass with `volumeBindingMode: WaitForFirstConsumer`** — critical. EBS volumes are AZ-pinned; `WaitForFirstConsumer` defers provisioning until a pod is scheduled, so the volume lands in the same AZ as the pod. Without this, a pod can land in AZ `us-east-1b` while its volume got provisioned in `us-east-1a`, and it just hangs forever.
- `reclaimPolicy: Retain` on the default class (don't let accidental PVC delete drop a database volume).
- Parameters: `type: gp3`, `encrypted: "true"`.
- **No cross-AZ storage replication.** For HA Postgres (CNPG), run 3 replicas across 3 AZs and let each get its own EBS volume in its own AZ — streaming replication handles HA at the database layer. Storage-level cross-AZ (Longhorn, Rook/Ceph) is unnecessary complexity for Postgres and inappropriate for most AWS workloads.

**StorageClass template:**
```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-gp3
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: ebs.csi.aws.com
volumeBindingMode: WaitForFirstConsumer
reclaimPolicy: Retain
parameters:
  type: gp3
  encrypted: "true"
```
