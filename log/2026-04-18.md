## Terraform Apply — humboldt Talos Cluster (Debugging)

First apply of `platform/humboldt/aws/` hit three separate blockers:

### Bug 1 — ASG tags with `/` rejected by AWS

`k8s.io/cluster-autoscaler/enabled` and `k8s.io/cluster-autoscaler/<cluster>` use `/` which AWS rejects
in propagated ASG tags (`propagate_at_launch = true`). The cluster autoscaler reads ASG tags directly
via the ASG API — it does not need them on instances. Fix: set both to `propagate_at_launch = false`
in `modules/talos/main.tf`.

### Bug 2 — Talos API SG only allowed Tailscale CGNAT range; subnet router SNATs

`talos_api_allowed_cidr = "100.64.0.0/10"` was correct in theory — Tailscale IPs should come from
that range. But Tailscale subnet routers perform SNAT when forwarding traffic into the VPC. The Talos
node sees source IP as the subnet router's VPC IP (`10.0.101.25`), not the originating Tailscale IP.
SG silently dropped all traffic → `i/o timeout` on port 50000.

Fix: changed `talos_api_allowed_cidr` (string) to `talos_api_allowed_cidrs` (list) in the module.
Passed `["100.64.0.0/10", "10.0.0.0/16"]` from `cluster.tf` — Tailscale CGNAT for direct future
access, VPC CIDR for subnet-router-proxied access today.

### Bug 3 — `tailscale up --accept-routes` silently ignored due to version mismatch

Mac had Tailscale app (tailscaled 1.96.5) already installed. Brew installed CLI 1.96.4. Running
`sudo tailscale up --accept-routes` via the brew binary against the app's tailscaled produced a
version mismatch warning and did NOT apply the setting — `tailscale status` showed:
`Some peers are advertising routes but --accept-routes is false`.

Fix: use the app's own binary directly:
```
sudo /Applications/Tailscale.app/Contents/MacOS/Tailscale set --accept-routes
```

Rule for future: when Tailscale app is installed on macOS, always use
`/Applications/Tailscale.app/Contents/MacOS/Tailscale` for CLI commands, not a separately
installed brew binary. Version mismatch = silent failures.

### Additional bugs fixed during apply

**Bug 4 — `talos_cluster_health` data source blocks every future apply**
The `data "talos_cluster_health"` resource runs on every `terraform apply` and waits for all
k8s nodes to report healthy (including worker ASG nodes). Workers take several minutes after
the ASG spins them up, making every apply hang for 5–10+ minutes. Removed from the module —
cluster health is better verified with `kubectl get nodes` after apply.

**Bug 5 — kubeconfig double base64 in Secrets Manager**
`talos_cluster_kubeconfig.kubernetes_client_configuration.ca_certificate` (and client_certificate,
client_key) already returns base64-encoded PEM. The module wrapped them in `base64encode()` again,
producing double base64. Kubeconfig was unreadable (`unable to parse bytes as PEM block`).
Fixed: removed `base64encode()` from the `aws_secretsmanager_secret_version` resource in
`modules/talos/main.tf`. The Python script used to build the initial kubeconfig decoded once
as a workaround.
