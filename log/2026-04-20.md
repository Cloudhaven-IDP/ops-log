>
> 1. **`Cloudhaven-IDP/Actions`** → push to `main`. Every `uses: Cloudhaven-IDP/Actions/...@main` in theo and theo-agents resolves from here. If this repo is empty on the org, all downstream CI fails immediately with "action not found".
> 2. **Terraform — CodeArtifact domain + repo** (`cloudhaven` / `python-private`) applied. Without it, `codeartifact-login` composite fails and no wheel can publish or be pulled.
> 3. **Terraform — 2 ECR repos** (`cloudhaven-dev`, `cloudhaven-prod`) applied. Without them, `docker-build-push-ecr` push step fails.
> 4. **Terraform — `gha-deployer` trust policy extended** to include `theo`, `theo-agents`, `theo-wearables`. Without it, OIDC `AssumeRoleWithWebIdentity` denies the runner.
> 5. **Terraform — `ci-bot` GitHub App** installed on `Cloudhaven-IDP` org; `CI_BOT_APP_ID` + `CI_BOT_PRIVATE_KEY` set as repo secrets on theo/theo-agents/theo-wearables. Without it, `gitops-bump-overlay` cannot commit or open PRs.
> 6. **theo-agents — first merge to main MUST be `libs/agent-core/` only.** This triggers `publish-agent-core.yaml` and publishes `agent-core>=0.1.0` to CodeArtifact. *Only after this wheel exists* can `nutrition/` or `meal/` Docker builds succeed (their `requirements.txt` installs `agent-core` from CodeArtifact at build time).
> 7. After step 6, subsequent theo-agents merges (nutrition, meal) build cleanly.
>
> **Dev previews and prod promotions will silently no-op or loudly red-CI if any of 1–6 is skipped. The chicken-and-egg bite is step 6 — easy to miss because `agent-core` is in-repo, but the container build pulls it from CodeArtifact, not the local path.**

### theo/ — path-filtered monorepo scaffolded

Three components, each its own pyproject/Dockerfile/tests subtree:
- `libs/theo-core/` — published **nowhere** (Theo-internal, editable install). Owns `config.py` (pydantic-settings), `dynamo.py` (typed accessors for chat/workout/sleep/feedback tables), `postgres.py` (SQLAlchemy session factory for cnpg), `s3_context.py` (markdown read/write), `memory.py` (Qdrant 1024-dim with bge-m3 embeddings from Ollama), `langfuse_client.py` (get_prompt / score / create_draft_prompt), `parlant.py`, `internal_http.py` (HMAC sign/verify with 60-second skew window), `logging.py` (structlog JSON with redaction).
- `api/` — thin FastAPI. `auth.py` verifies an HMAC on the Envoy-injected `X-Soma-User-Id` header (Clerk JWT never reaches the app — the gateway has already verified it via `SecurityPolicy`). Routes: `/chat` forwards a signed payload to the orchestrator; `/feedback` writes to DynamoDB and calls `langfuse.score()`; `/proactive/dispatch` is the worker callback; `/health` + `/ready`.
- `orchestrator/` — LangGraph `StateGraph(ConversationState)` with nodes `classify_intent → inject_memory → load_user_context → dispatch → apply_persona`. Coach persona fetches the master prompt from Langfuse with a 5-minute TTL cache. Onboarding checklist is **also** fetched from Langfuse (prompt name `theo-onboarding-checklist`, JSON body) — adding/editing onboarding questions = Langfuse UI edit, no redeploy. No hardcoded tuples.
- `worker/` — Temporal worker registers six workflow classes (session_loader, nightly_summarize, weekly_report, daily_brief, proactive_nudge, prompt_judge_daily) as signatures + docstrings. Real bodies land later.

Only real test this pass: HMAC sign/verify round-trip in `libs/theo-core/tests/test_internal_http.py` — covers happy path, body tampering, expired timestamp, wrong secret. The `/health` endpoint test is there mainly to exercise middleware wiring.

**Rule:** no tautological tests in scaffolds. `assert len(_WORKFLOWS) == 6` tests the test, not behavior; deleted. A pure-wiring module gets no test file until it has real logic.

**Rule:** onboarding checklists, dispatch tables, persona prompts — anything that a non-engineer should be able to tweak — lives in Langfuse (or S3 for longer docs), not in a Python tuple. Fetch with TTL cache at startup so promotions propagate without redeploy.

### theo/.github/workflows/ — DRY matrix CI + per-env deploy

Three workflows, all thin callers of the Actions repo:

- `ci.yaml` — one `changes` job runs `dorny/paths-filter@v3`; one `lint-test` job matrixes over `fromJSON(needs.changes.outputs.components)` and calls `python-ci`. Filter names (`api`, `orchestrator`, `worker`) are deliberately the same as subdir names so `matrix.component` drops straight into `workdir`. Adding a component = one filter edit.
- `build-and-deploy-dev.yaml` — triggered on `pull_request`. `prep` job computes short-sha + overlay path. Matrix `build-push` → `docker-build-push-ecr`. Matrix `bump-gitops` → `gitops-bump-overlay` (dev overlay path = `overlays/dev/pr-<N>/theo/kustomization.yaml`). Because the trigger is `pull_request`, the reusable workflow's single branching step direct-commits to main in the gitops repo (dev autosync picks it up).
- `promote-to-prod.yaml` — triggered on `push` to main. `changes` job runs paths-filter against `github.event.before..github.sha` so only the components that actually changed in the merge commit get promoted. Matrix `crane-copy` dev→prod. Matrix `bump-gitops-prod` opens a PR via ci-bot (again, the reusable workflow sees `push` and flips to PR mode). ArgoCD sync for prod will fire from a separate workflow **inside the GitOps repo** after the PR merges — not this workflow.

**Rule:** for a path-filtered monorepo, never write N per-component jobs. One matrix job driven by the paths-filter `outputs.changes` JSON array. Filter name == subdir name.

**Rule:** when a workflow's two behaviours map cleanly to `pull_request` vs `push`, drop the `mode:` input and branch on `github.event_name` in one shell step. Fewer footguns, no redundant input to keep in sync.

**Rule:** every workflow step needs a descriptive `name:`, and step IDs are real words (`image-tag`, `ecr-login`, `codeartifact`), not 2-letter mnemonics. The GHA UI heading is the first thing someone reads when a job fails at 2am.

### Late-evening third pass — Postgres platform plan landed

Plan: [`ops-log/plans/2026-04-20-postgres-platform.md`](../plans/2026-04-20-postgres-platform.md). Reframed from `~/Downloads/postgres-platform-context.md` (architecture brain-dump → blog series source → actionable cut). Wiring starts tomorrow with Phase 0.

**The shape of the design:**
- One Helm chart at `gitops/charts/postgres-platform/`, values files keyed on **storage class** (`values.local-path.yaml`, `values.ebs-gp3.yaml`) — not on cluster name. New cluster on a known storage backend = no new values file.
- One TF module at `Infrastructure/modules/postgres-platform/` (Afolabi authors), uses `cyrilgdn/postgresql` provider, consumes the same `app_roles` shape as the chart.
- Per-tenant Cluster CRs (`soma-pg`, `temporal-pg`, `langfuse-pg`) — three separate clusters, three SSM path prefixes, three DNS aliases (`soma.internal`, `temporal.internal`, `langfuse.internal`). No shared blast radius.

**Auth split — the call after several iterations:**
- **Apps: cert-based, passwordless.** CNPG runs its own CA → cert-manager `ClusterIssuer` → per-`appRoles` `Certificate` with `commonName` == PG LOGIN role. `pg_hba`: `hostssl ... cert clientcert=verify-full`. Apps mount `/certs/{tls.crt,tls.key,ca.crt}`, connect with `sslmode=verify-full`. Auto-rotation via cert-manager (90d duration, 30d `renewBefore`).
- **Master: password in SSM.** CNPG-generated superuser → ESO `PushSecret` → AWS SSM SecureString as JSON `{username, password}`. CA cert pushed separately as a non-sensitive SSM `String`. Terraform reads via OIDC, connects as `postgres` with username/password. *Earlier draft used a `terraform_admin` cert + `pg_ident` map; killed — too much plumbing for one connection. Cert-auth earns its keep when there are many consumers; the master is one credential.*
- **Humans: out of scope for v1.** No SSO, no Authentik, no CloudBeaver. Future plan: JIT cert-issuance system minting 1h-TTL certs tied to human identity.

**Role model:** NOLOGIN group roles per `(database, schema, mode)` triple (`soma_theo_rw`, `soma_theo_ro`, `soma_theo_admin`) + LOGIN cert roles inheriting membership. `ALTER DEFAULT PRIVILEGES` so future tables get correct grants. Adding an app = create cert + `GRANT <group> TO <login>`. Two lines.

**Secret backends:** `secretBackend.type: ssm | secretsmanager | vault`. SSM default (free at our volume). SM supported. Vault is a stub.

**Phasing:** P0 (cnpg + cert-manager + `ebs-gp3` SC on humboldt) → P1 (cluster only, no certs/DNS/ESO) → P2 (CoreDNS rewrite) → P3 (app cert auth, one test role) → P4 (ESO PushSecret superuser → SSM) → P5 (TF module, password path) → P6 (multi-tenant: temporal + langfuse) → P7 (Nebulosa overlay).

**Open questions parked for tomorrow:** Talos durability of the CoreDNS ConfigMap patch (§9.1), `ClusterIssuer` cross-namespace CA Secret access (§9.2), and confirming the SSM password-rotation drill works end-to-end before relying on it for prod.

**Rule:** values files keyed on infra primitive (storage class), not cluster name. Anyone with the same storage backend benefits from the same overrides — no `values.<cluster>.yaml` proliferation.

**Rule:** cert-auth earns its keep with many consumers (auto-rotation, no shared secret, ergonomic). For one admin credential, password-in-SSM is the right shape — don't over-engineer the rare connection.

**Rule:** rotation must not require a redeploy. cert-manager handles app certs; deleting the CNPG-generated superuser Secret triggers regeneration + ESO re-push. App pods don't restart for either.
