## Blog Idea — Scale-to-Zero ML Models on Kubernetes

**Angle:** Lambda-like economics on owned hardware. The key insight nobody talks about: you don't have to choose between paying for idle GPU memory and paying cloud inference bills. KubeAI's scale-from-zero gives you a third path.

**The pattern:**

```yaml
apiVersion: kubeai.org/v1
kind: Model
spec:
  minReplicas: 0   # costs nothing at idle
  maxReplicas: 4   # scales under load
  resourceProfile: gpu:1
```

Request hits KubeAI proxy → buffered → 0→1 scale triggered → model loads → request served. No dropped requests.

**Tradeoff vs Lambda:** cold start is 30s–5min depending on model size vs Lambda's ~100ms. Acceptable for non-latency-SLA workloads (personal coaching summaries, meal planning, weekly reports).

**Resource efficiency story:**
- 70B model at zero replicas = 0GB RAM used (vs 40GB+ when loaded)
- Embedding model stays warm at `minReplicas: 1` — it's small and latency-sensitive (every RAG lookup)
- Heavy models (coaching summary, weekly report) at `minReplicas: 0` — triggered by APScheduler jobs or user requests
- Scale back to zero after idle timeout

**Event-driven horizontal scaling:** KubeAI watches its internal request queue depth and scales replicas accordingly. 100 requests queued → scale to 2 pods → 50 requests each. The built-in load balancer distributes using CHWBL (prefix-hash-aware, keeps KV cache warm across replicas). KEDA can also drive scaling from external signals (SQS depth, Prometheus metrics) if finer control is needed.

**Minisforum angle:** Ryzen AI Max+ 395 (Strix Halo) + 128GB LPDDR5X at ~$800–1,500. Run `llama3.1:70b-q4` locally at Lambda economics — zero when idle, on-demand when needed. Changes the Bedrock cost calculus significantly.

**Post structure:**
1. The problem: GPU memory is always-on, cloud inference is per-token
2. KubeAI scale-from-zero explained
3. What cold start actually feels like at different model sizes
4. The embedding exception (always-warm)
5. KEDA for queue-driven horizontal scale
6. Real numbers: RAM saved, Bedrock cost avoided, cold start latency

## Blog Idea — Pre-Scaling LLM Pods with APScheduler + KEDA Before Load Hits

**Angle:** Scale-from-zero is great until your scheduled coaching jobs all fire at 06:00 and every pod is cold. The fix: KEDA reads the SQS queue depth *before* the workers process it, pre-warming pods so there's no cold start on predictable workloads.

**The pattern:**

```
APScheduler fires 06:00 coaching batch
  → drops 50 jobs into SQS
  → KEDA sees queue depth spike
  → scales pods from 0 → N before first job is dequeued
  → jobs process warm, no cold start penalty
```

```yaml
triggers:
  - type: aws-sqs-queue
    metadata:
      queueURL: https://sqs.us-east-1.amazonaws.com/.../coaching-jobs
      targetQueueLength: "10"  # 1 pod per 10 queued jobs
```

**Why this matters:** Scale-from-zero only works painlessly when you can absorb cold starts. For user-facing requests, 30–60s is visible. For batch scheduled work (morning briefing, weekly summary), KEDA can pre-scale so by the time the first job is dequeued the pod is already warm.

**Combined with KubeAI:** KubeAI handles the 0→1 reactive scale for on-demand requests. KEDA handles the 0→N proactive scale for batch workloads. Two different scaling signals, same pod fleet.

**Post structure:**
1. The cold start problem for scheduled LLM workloads
2. How KEDA reads external signals (SQS, Prometheus, custom metrics)
3. APScheduler → SQS → KEDA pre-scale flow
4. KubeAI reactive + KEDA proactive: combining both
5. Real numbers: latency before vs after, cost of keeping N pods warm vs on-demand

### `afolabi-next/.github/workflows/`

**`ci.yml`** (workflow name: `preview`) — single file, two modes driven by `github.event_name`:

| Trigger | Jobs that run |
|---|---|
| `pull_request` | `ci` only (lint, typecheck, build) |
| `push: main` | `ci` → `deploy-preview` → `gitops-bump-preview` |

Preview image pushed to `cloudhaven-dev` ECR as `afolabi-next-{sha7}`. Gitops overlay bumped at `deployment/humboldt/preview/afolabi-next.yaml` — a separate file from prod so ArgoCD treats preview as its own short-lived Application.

**`release.yml`** — production promotion:

Triggers: `workflow_dispatch` (manual, any time) or `push: tags: ['v*']` (automated on git tag).

`workflow_dispatch` inputs:
- `preview-tag` — which preview build to promote (defaults to latest main sha)
- `release-tag` — semantic version for prod (defaults to pushed git tag)
- `sync` — boolean, default `true`

Job chain:
```
resolve → promote (crane-copy dev→prod) → gitops-bump-prod → cleanup-preview
                                        ↘ argocd-sync (skipped only on dispatch + sync:false)
```

- `resolve`: derives all refs. Preview tag `afolabi-next-{sha7}` → prod tag `afolabi-next-v1.2.3`. Both full ECR refs computed here, referenced downstream via `needs.resolve.outputs.*`.
- `promote`: crane-copy. No rebuild — promotes the exact preview image that passed CI.
- `gitops-bump-prod`: bumps `deployment/humboldt/afolabi-next.yaml` (the default path via `cluster: humboldt`).
- `cleanup-preview`: checks out GitOps repo, `git rm deployment/humboldt/preview/afolabi-next.yaml`, pushes to main. ArgoCD notices the file is gone and prunes the preview Application + namespace.
- `argocd-sync`: skipped only when `workflow_dispatch` with `sync: false`. Always runs on tag push.

**Rule:** dev environment is a preview namespace, not a persistent environment. `cloudhaven-dev` ECR holds short-lived preview images; `cloudhaven-prod` ECR holds promoted release images. The preview overlay is ephemeral — created on every push to main, deleted on every prod release.

**Rule:** `gitops-bump-overlay` default path is `deployment/{cluster}/{image-name}.yaml`. Override with `overlay-path` for non-standard cases (preview, multi-cluster, etc.).

## `/ask` — lightweight identity layer

Small auth gate on the chat UI so visitors leave a name + email before talking to the agent. Three wins in one change:
1. Real contact info for follow-up / outreach
2. Email-keyed rate limiting in the Cloudflare Worker (cleaner than IP on shared networks)
3. Personalization hook for the agent's system prompt

**Not** a modal blocker, not OAuth, not a backend session — just a small form at the top of `/ask` with `localStorage` persistence and a cookie that the Worker can read. Identity can be edited inline ("as Afolabi · email [edit]").

### Files touched

**`afolabi-next/src/components/visitor-identity.tsx`** (new):
- Two inputs (name, email) + Continue button. Compact, matches the editorial design.
- Toggles between edit mode and a one-line "as {name} · {email} [edit]" display.
- Client-side email validation (simple regex).

**`afolabi-next/src/app/ask/page.tsx`**:
- `visitor` state, loaded from `localStorage` on mount.
- `saveVisitor()` writes to `localStorage` AND sets `cf_visitor` cookie with base64-encoded email (max-age 30 days, `SameSite=Lax; Secure`).
- `send()` gated by `!visitor` — textarea disabled, Send button disabled, seed-prompt buttons disabled until identity is set.
- `?q=` param flow updated: if a pending query arrives before identity is set, it's stashed and auto-sent after the user fills the form.
- Seed prompts cut from four generic questions to two sharper hooks: *"Tell me something the résumé doesn't."* and *"Does Afolabi hate the cloud?"*

**`afolabi-next/src/app/api/ask/route.ts`**:
- `parseVisitor()` validates name (1-120 chars) and email (regex + length cap).
- Returns `400 "identity required"` when visitor is missing or malformed.
- Forwards to agent as `X-Visitor-Name` / `X-Visitor-Email` headers *and* in the JSON body (belt-and-suspenders; agent can read either).

**`Infrastructure/applications/afolabi-agent/general/files/index.ts`** (Worker):
- Reads `cf_visitor` cookie, base64-decodes to email, lowercases.
- Rate-limit key is `v:email:<email>` when cookie present, `v:ip:<ip>` otherwise.
- Same 15 req / 15 min per-visitor window, same 200/hr global cap.
- Cookie parsing is inline — no library, ~10 lines.

### Design choices worth remembering

**Rule:** client is the single source of truth for identity. `localStorage` persists across visits, cookie exists only so the Cloudflare Worker can see the email for rate limiting. No server-side session.

**Rule:** cookie contains *just* the email (base64), not the full visitor object. Smaller payload, less surface area, Worker doesn't need to parse JSON.

**Rule:** the gate is a soft inline form, not a modal. Modals feel aggressive for a portfolio site; an inline row feels like an editorial subscription prompt — in-tone with the rest of the UI.

**Rule:** proxy sends identity via both headers and body. The agent build session can decide which to read from — neither locks them in.

### Deferred to outreach-agent session

- DynamoDB tables for lead capture (`afolabi-agent-visitors`, `afolabi-agent-conversations`).
- IRSA policy for DynamoDB write access (will use the new Bedrock-policy-module pattern — a DynamoDB-policy module attached to `app_role.policy_arns`).
- Agent-side persistence of conversations with visitor email as partition key.
- Outreach agent (separate service) that reads conversations, classifies intent (recruiter / engineer / other), summarizes, and surfaces high-signal leads.

This means the current identity data is ephemeral — the visitor's info is known to the agent via headers, but not persisted anywhere yet. Lead capture is one Terraform apply + one agent code change away.

## `/ask` — second-pass fixes after live testing

Live-testing the identity layer surfaced three issues worth fixing on the spot:

### 1. Home-page `AskInline` lost the query in a black hole

Flow before: type question → click Ask → redirect to `/ask?q=…` → `/ask` sees no visitor → shows form → question invisible behind the form, user thinks it vanished.

Fix: collect identity on the home page too. `AskInline` now checks `localStorage` on mount. If no visitor, the component renders name + email inputs above the question bar — all three required before submit. On submit, identity saved first, *then* redirect. By the time `/ask` loads, visitor is already in `localStorage`, the pending query auto-sends, no empty state.

Returning visitors on the home page just see the single question bar with a subtle `as {name} · {email}` line underneath.

### 2. `VisitorIdentity` lingered on `/ask` for returning visitors

Bug: component's internal `editing` state was initialized via `useState(!visitor)`, capturing the initial `null` before `loadVisitor()` ran. When visitor became set, `editing` stayed `true` and the form never collapsed.

Fix: simplified `VisitorIdentity` to a pure form (always in edit mode, no internal toggle). The parent (`ask/page.tsx`) owns the visibility decision — renders the form only when `!visitor || editingIdentity`. Parent also owns the edit escape hatch in the fixed bottom bar: `as {name} · edit` underline, click to re-open the form pre-filled, with a `cancel` button to back out without saving.

### 3. Shared helpers extracted to `src/lib/visitor.ts`

Three consumers now (`AskInline`, `VisitorIdentity`, `/ask/page.tsx`) so the `Visitor` type + `loadVisitor()` / `saveVisitor()` / `isValidEmail()` moved to one module. No more inline duplicates.

### Seed prompts trimmed

Four generic questions → two sharper hooks that invite a real answer: *"Tell me something the résumé doesn't."* and *"Does Afolabi hate the cloud?"*

## Cloudflare Worker — layered rate limiting against cookie-swap bypass

Live-test question: *if I clear my cookie and set a different email, do I get a fresh 15 requests?* Yes, under the first implementation. That's a real bypass for anyone who reads the JS.

Rewrote the Worker with three layered limits — every request checks all three, first one that trips wins:

| Layer | Limit | Key | What it catches |
|---|---|---|---|
| Global | 200 / hour | `g:<hour>` | Total traffic safety belt (protects model spend) |
| Per-IP burst | 20 / minute | `b:<ip>:<minute>` | Cookie-swap bypass; rapid-fire abuse. Keyed on `CF-Connecting-IP`, not user-controlled |
| Per-visitor | 15 / 15 min | `v:<email>` | Friendly granular limit per identified user. Only applies when cookie is present |

All three KV checks run in parallel via `Promise.all`. All three writes also parallel, via `ctx.waitUntil` so they don't block the response.

**Rule:** user-controlled identifiers (cookies, headers the client sets) are for personalization and granular limits, never the hard ceiling. The hard ceiling always keys on something the client can't spoof — for Cloudflare workloads that's `CF-Connecting-IP`, normalized at the edge.

**Rule:** layer rate limits by trust level. Global (trust nothing) → IP (trust Cloudflare's edge view) → visitor (trust the cookie but only for friendly limits). An attacker would need a residential IP pool to defeat layer 2, orders of magnitude more effort than clearing cookies.

Tunable constants at the top of `index.ts` (`PER_IP_BURST_MAX`, `PER_VISITOR_MAX`, `GLOBAL_MAX`). Stricter values need no redeploy logic change, just a Wrangler push.

Chart now renders `hostssl langfuse langfuse all md5` in `pg_hba`.

### Caveats shipped as chart-level comments / defaults, not as runtime surprises

- **Namespaced `Issuer`, not `ClusterIssuer`.** cert-manager resolves `issuerRef.kind: Issuer` inside the Certificate's namespace. For Langfuse (app + Cluster both in `langfuse`), this works. When theo/theo-agents come online — Cluster in `soma-pg-system`, Certificate in `theo` — we need either (a) `ClusterIssuer` + the `<cluster>-ca` Secret copied into cert-manager's `--cluster-resource-namespace`, or (b) per-app-namespace `Issuer` + `trust-manager` `Bundle` to distribute the CA. Plan §9.2 open question; deferring the call until theo lands.
- **`secretBackend.type: none` default; ESO templates render empty.** Flipping to `ssm` requires the Infrastructure repo to ship the IRSA role first (per-tenant ESO role with `ssm:PutParameter` on `<tenant>/postgres/*`). Until then, the superuser Secret stays in-cluster only.
- **`coredns.rewrite.enabled: false` default.** Talos reconciliation durability of the CoreDNS ConfigMap patch is Plan §9.1 open question. Enable per tenant after verifying on a test cluster that a `talosctl machineconfig apply` doesn't revert our patch.
- **`monitoring.enablePodMonitor: false` default.** Prometheus Operator CRDs aren't installed on humboldt yet. Flip on after VictoriaMetrics / kube-prometheus-stack lands.

### Rules learned in the second pass

- **Don't half-implement a plan just to land a feature faster.** The "short path" skipped five platform templates the plan explicitly scoped. Langfuse would have run, but every future tenant (theo, temporal) would hit the same gaps. Writing the templates now (defaults-off, runtime-gated) means future tenants flip flags, not add code.
- **ESO PushSecret from CNPG's superuser Secret → two SSM params beats one JSON blob.** Terraform consumer reads each parameter by name — no `jsondecode`, no schema drift if CNPG changes the Secret shape. Same total SSM cost (both free tier).
- **Hook-sequenced Jobs with `alpine/k8s:1.30.0` are the right shape for ConfigMap patches in Helm.** kubectl + awk in one image, no python/jq sidecar dance. Marker-comment idempotency is clean; pre-delete stripping is the same Job logic in reverse.
- **`Issuer` vs `ClusterIssuer` is a deployment-topology question, not a convenience pick.** Single-namespace (Langfuse) → `Issuer`. Cross-namespace fan-out (theo) → `ClusterIssuer` + CA Secret placement strategy. Don't pick `ClusterIssuer` "to be safe" — it hides the CA-secret-namespace decision.

| EBS gp3 | small | ~100 GiB | | ~$8 |
| **All-in on-demand** | ~$75 | | | **~$179/mo** |

CP stays tainted — we do *not* schedule workloads on it. The 4 GiB upgrade from `t3.small` is purely for etcd comfort; `t3.small`'s 2 GiB is tight enough to throw `etcdserver: apply took too long` warnings under modest load.

**ASG: `min:1 / max:3 / desired:1`.** Scale up during upgrades or unexpected load; default lean.

### Compute Savings Plan

Decided: **1-year no-upfront Compute Savings Plan, commit $0.16/hr (~$117/mo).** Covers the current shape at ~100%.

- **Why Compute SP not EC2 Instance SP:** humboldt shape is still evolving (t3 → r6g this week, r6g.xlarge → r6g.2xlarge in 2–3 months). Compute SP covers any family / size change; EC2 Instance SP locks to `r6g` family only. ~10% extra discount on EC2 Instance SP isn't worth the flex loss here.
- **Why 1-year not 3-year:** too early to commit 3-year when the cluster shape is changing month-over-month.
- **Why no-upfront not all-upfront:** ~5–7% more discount isn't worth the cash outlay for a personal project; monthly billing matches how the rest of the spend is already shaped.
- **Stacking later:** when the worker bumps to `r6g.2xlarge`, buy a second Compute SP at ~$0.14/hr on top. SPs are additive; no cancellation needed.

**Effective all-in after SP purchase: ~$125/mo.** Savings vs on-demand: ~$54/mo, ~$648/yr.

**Order of operations matters:** apply the `cluster.tf` instance-type change first, let the new ASG stabilize for ~1h, *then* buy the SP. Purchasing while still on `t3.medium` wastes the commit on instances about to be terminated.

### ARC runners + Spot — plan + issue opened

Fargate was the obvious scale-to-zero answer for GHA runners, but **Fargate is EKS-only and humboldt is Talos on EC2** — hard incompatibility. Chose instead: **ARC v2 + Karpenter + arm64 Spot on humboldt itself**. Matches Fargate ergonomics without a second cluster.

Full plan at [`ops-log/plans/2026-04-22-arc-runners-spot.md`](../plans/2026-04-22-arc-runners-spot.md). Tracking issue on the Infrastructure repo: [Cloudhaven-IDP/Infrastructure#3](https://github.com/Cloudhaven-IDP/Infrastructure/issues/3).

Highlights:
- Karpenter `NodePool` `ci-spot-arm64`, taint `ci-only:NoSchedule`, `limits.cpu: 32` hard cap, Bottlerocket AMI family (Karpenter has no Talos AMI support today).
- ARC v2, GitHub App on the `Cloudhaven-IDP` org, App creds → Secrets Manager → ESO.
- Custom runner image in `cloudhaven-dev` ECR, baked with `docker buildx`, `crane`, `kubectl`, `aws-cli v2`, `tailscale`, `gh`. Weekly CronJob rebuild.

- Tailscale sidecar, ACL `tag:ci-runner → tag:humboldt-argocd:443` only.
- Expected cost: <$10/mo in Spot + negligible ECR. The value is capability (IRSA + Tailscale + native arm64), not dollar savings.

### Rules learned

- **Right-size at the moment of need, not preemptively.** Bumping `worker_instance_type` in an ASG is a 5-min rolling refresh. The cost of over-provisioning for 2 months to avoid a 5-min change at month 3 is ~$120/mo. Don't pay it.
- **Compute SP over Standard RI / EC2 Instance SP when the shape is still evolving.** The ~10% extra discount on locked-family plans isn't worth giving up the ability to change family, size, or region during the term. Only commit to narrower plans once the shape is boring.
- **Order for a shape change: TF apply → wait for stability → buy SP.** Buying the SP while the old instance type is still running wastes the commit on instances about to terminate.
- **Fargate + Talos is an incompatibility trap.** Anyone reading "Fargate is serverless Kubernetes" thinks it drops into any cluster. It doesn't — it's EKS-only, EKS-control-plane-managed. If you're not on EKS, Fargate isn't in your toolbox.
- **ARC v2 + Karpenter + Spot is the "Fargate-equivalent" on non-EKS clusters.** Scale-to-zero ergonomics, pay-per-use compute, no second cluster.

### Walking back the phased sizing — deploy sprint changed the math

Update from mid-session: the "r6g.xlarge now, r6g.2xlarge in 2–3 months" phased plan assumed Temporal + theo + observability would land over months. Reality: **all of it lands this week.** Phased path saves ~$4/yr in that timeline while forcing an ASG refresh + second SP purchase during the busiest deploy sprint. Killed.

**Final shape bought:** `t4g.small` CP (tainted) + `r6g.2xlarge` × 1 worker, **ARM rebuild** (not x86 in-place). Compute Savings Plan purchased at **$0.30/hr, 1-year no-upfront**.

**Why the ARM rebuild won over "x86 in-place resize":**
- Module's AMI lookup hardcodes the arch, and CP is a direct `aws_instance` not an ASG — so even an x86-only CP resize is a replace (etcd state lost). Once the rebuild is unavoidable for the CP, the worker arch change is free.
- Current humboldt has ~nothing stateful (CNPG not yet installed, no workload PVCs). Rebuild cost is 30–60 min of re-applying K8s-Bootstrap dirs + re-syncing ArgoCD. Lowest it'll ever be.
- $0.30/hr SP is priced against ARM rates. Staying x86 would have wasted ~17% of the SP commit (Compute SP discount on x86 r6i is ~24% vs ~30% on ARM r6g). ~$51/mo SP-mismatch premium over the year ≈ **$612/yr** lost for keeping a cluster that was 95% empty.

### Module change: `modules/talos/` now parameterizes arch

The module's `data.aws_ami.talos` regex was hardcoded `-amd64$`. Added a `var.architecture` (default `arm64`, values: `arm64 | amd64`, validated) and threaded it through the regex. Existing single caller (humboldt) inherits the new default. Future x86 callers pass `architecture = "amd64"` explicitly.

**Rule:** module defaults should match the forward architecture, not legacy x86. Graviton is the performance- and cost-optimal default for new clusters; if a caller needs Intel, that's the explicit override.

### Cutover runbook for the rebuild

Six-phase runbook committed to the session:
1. `terraform apply` in `Infrastructure/platform/humboldt/aws/` → CP + worker replace, ~5–10 min
2. Fetch new kubeconfig from Secrets Manager (`humboldt/kubeconfig`)
3. Apply `K8s-Bootstrap/humboldt/*` in order: gateway-api → ebs-csi-driver → cert-manager → external-secrets → cnpg-operator → envoy-gateway → tailscale-operator → argocd
4. `terraform apply` in `Infrastructure/platform/humboldt/cloudflare/` + `.../argocd/` → re-bind kubernetes provider to new cluster, re-create ClusterRoleBinding + ArgoCD AppProjects
5. Pre-create `langfuse-secrets` in the `langfuse` namespace (three `openssl rand`-generated secrets: salt, encryption-key, nextauth-secret)
6. Apply `gitops/apps/postgres-langfuse/argocd-app.yaml` + `gitops/apps/langfuse/argocd-app.yaml` — ArgoCD syncs the CNPG Cluster CR + Langfuse + in-chart dependencies (Valkey, ClickHouse, MinIO)

What survives the rebuild: VPC/subnets/SGs/IAM, Talos machine secrets (persisted in TF state → same cluster identity from Talos's POV), OIDC issuer URL (same Cloudflare tunnel cert → same AWS IAM OIDC thumbprint → IRSA trust policies unaffected), Secrets Manager entries (re-populated on apply).

What dies: all K8s state (etcd on old CP root volume), all pods, old CP private IP. Cloudflare tunnel targets are cluster-DNS service names (`kubernetes.default.svc`, `argocd-server.argocd.svc`), not IPs — no tunnel re-config needed.

### Rules learned (post-decision)

- **"In-place resize" is a lie when the CP is a single `aws_instance` direct resource.** Talos with 1 CP + module-as-written = any CP instance type change is a destroy/create + etcd wipe. The only way to preserve etcd across a single-CP arch change is an etcd snapshot → new CP → restore dance, which for a nearly-empty cluster is more effort than a rebuild.
- **Compute SP rates differ materially by arch (~6% delta between ARM and x86 equivalents).** When buying an SP, commit rate should match the arch you'll actually run — or you burn the commit on a discount bracket you can't access.
- **Rebuild the cluster while it's empty, not when it's full.** State accumulates fast once workloads start landing. The cost of a rebuild at T+0 is minutes; at T+3 months it's "migrate three postgres clusters, three vector stores, three sets of PVCs, and a live Langfuse UI."

## Rebuild execution — what actually happened

`terraform apply` went through in `Infrastructure/platform/humboldt/aws/`. Cluster came back as `t4g.small` CP + `r6g.2xlarge` worker, both arm64, Talos v1.12.6, Kubernetes v1.35.0. Compute SP purchased at $0.30/hr 1-yr no-upfront. **But three module bugs surfaced during the cutover and each needed a manual intervention.** Capturing so the next rebuild is smoother.

### Module bug 1 — `talos_machine_bootstrap` is a one-shot, doesn't re-run on CP replace

TF apply output showed:
```
module.talos.talos_machine_bootstrap.this: Modifying... [id=machine_bootstrap]
module.talos.talos_machine_bootstrap.this: Modifications complete after 0s [id=machine_bootstrap]
```

"0s modification" = no-op — TF state already had `machine_bootstrap` as applied, so it didn't re-invoke bootstrap on the new CP. Result: new CP came up with empty etcd, kubelet waiting on apiserver, apiserver waiting on etcd, `talosctl service etcd` showing `Health check failed: context deadline exceeded`.

Port 6443 refused; port 50000 (Talos API) open — the split told us apiserver wasn't running, but everything else was fine.

**Manual fix executed:**
```bash
# 1. pull talosconfig from TF state (since the root didn't re-export module outputs, and terraform output -raw
#    injects ANSI codes that break yaml parsers — using terraform state pull + python was the clean path)
terraform state pull | python3 -c "... extract talos_client_configuration.this.attributes.talos_config ..."

# 2. bootstrap failed: "etcd data directory is not empty" — new CP had a partial etcd dir from first (failed) start
talosctl reset --graceful=false --system-labels-to-wipe=EPHEMERAL --reboot

# 3. after reboot, bootstrap succeeded
talosctl bootstrap

# 4. ~30s later, 6443 open, kubelet + apiserver + flannel + coredns came up clean
```

**Fix for the module:** add `replace_triggered_by = [module.control_plane[0].aws_instance.this]` (or similar) to the `talos_machine_bootstrap` lifecycle so it re-runs on CP replacement. Deferred — not blocking today.

### Module bug 2 — ASG has no `instance_refresh` block

Changing `worker_instance_type` updated the launch template to `$Latest`, but the running `t3.medium` instance stayed alive. AWS doesn't auto-replace ASG instances on LT changes without an explicit `instance_refresh` block.

**Manual fix executed:**
```bash
aws autoscaling start-instance-refresh --auto-scaling-group-name humboldt-workers \
  --preferences '{"MinHealthyPercentage":0,"InstanceWarmup":60}'
```

`MinHealthyPercentage: 0` required because `desired=1` can't keep 100% healthy while replacing the only node.

**Fix for the module:** add to `aws_autoscaling_group.workers`:
```hcl
instance_refresh {
  strategy = "Rolling"
  preferences {
    min_healthy_percentage = 0
    instance_warmup        = 60
  }
  triggers = ["launch_template"]
}
```
Deferred.

### Module bug 3 — `humboldt/kubeconfig` SM secret has stale `kubeconfig_raw` while structured fields are correct

After apply, SM showed:
```
host: https://10.0.101.9:6443          ← NEW CP (correct)
kubeconfig_raw.server: https://10.0.101.165:6443   ← OLD CP (stale YAML blob)
```

Downstream TF consumers (cloudflare, argocd) read `host` via `jsondecode()` + field access, so they saw the correct IP. But anything pulling `kubeconfig_raw` directly (the path I first tried) got the dead old IP → `dial tcp 10.0.101.165:6443: connect: network is unreachable`.

**Manual fix executed:** `sed` replace the server line in the locally-saved kubeconfig, then `kubectl config set-cluster humboldt --server=https://10.0.101.9:6443` after merging into `~/.kube/config` (kubectl merge keeps the first-occurrence cluster entry on name conflict — old stale entry from a previous session won).

**Root cause:** TF's `talos_cluster_kubeconfig` regenerated `kubeconfig_raw` using a Talos endpoint value that was captured before the CP replacement finalized. Fix unclear — possibly forcing `talos_cluster_kubeconfig` to depend on the new CP's IP output, or moving to the structured field consumers only and dropping `kubeconfig_raw` from SM. Deferred.

### Additional chore — kubectl merge doesn't overwrite conflicting cluster entries

`KUBECONFIG=~/.kube/config:~/.kube/humboldt.kubeconfig kubectl config view --flatten` kept the **first** cluster entry with name `humboldt` (the stale one from old `~/.kube/config`), not the new one. Had to `kubectl config set-cluster humboldt --server=...` explicitly after the merge. Minor, file under "kubectl footguns."

### The IRSA blocker — parked, unresolved

`ebs-csi-controller` Deployment is CrashLoopBackOff (16+ restarts, 1/5 containers ready). Diagnosis:

- OIDC issuer `https://oidc-humboldt.cloudhaven.work` returns HTTP 200 on both `.well-known/openid-configuration` and `/openid/v1/jwks`. AWS IAM OIDC provider registered and valid.
- `humboldt-ebs-csi-controller` IAM role's trust policy is **strictly `sts:AssumeRoleWithWebIdentity`** from the humboldt OIDC provider, restricted to `system:serviceaccount:kube-system:ebs-csi-controller-sa` — cannot be assumed via instance profile.
- SA `ebs-csi-controller-sa` has the `eks.amazonaws.com/role-arn` annotation set correctly.
- **Pod spec has no `AWS_ROLE_ARN` / `AWS_WEB_IDENTITY_TOKEN_FILE` env vars, no `sts.amazonaws.com`-audience projected token volume.**
- Worker EC2 instance profile: `None`. IMDS returns 404 for `/iam/security-credentials/`.
- Cred chain: empty env → no IRSA env vars → IMDS 404 → crash.

**Three paths out, none taken tonight:**
1. Install `amazon-eks-pod-identity-webhook` as a new K8s-Bootstrap dir. Mutating webhook injects IRSA env vars + projected token on every pod whose SA has the annotation. Scales to all future AWS-calling workloads (external-secrets SSM push, CNPG PushSecret, theo-agents → Bedrock/DynamoDB/SSM, and every other IRSA workload planned). This is the standard answer for non-EKS Kubernetes clusters doing IRSA; historical claim of "we never needed this" was a latent bug from never exercising IRSA on the old cluster.
2. Create a separate IAM role trusted by `ec2.amazonaws.com`, attach `AmazonEBSCSIDriverPolicy`, attach as instance profile to worker EC2s. Broader blast radius (role serves whole node, not one SA), but bypasses webhook.
3. Manually patch the `ebs-csi-controller` Deployment with the projected token volume + IRSA env vars. One-off, doesn't scale to the other workloads.

**Parked as an open question.** Will resolve before attempting the first CNPG Cluster CR sync (since CNPG needs EBS CSI working to bind PVCs).

### Rules learned

- **Talos module's `talos_machine_bootstrap` resource doesn't re-trigger on CP replacement.** After any CP instance-type or AMI change, manually: `talosctl reset --graceful=false --system-labels-to-wipe=EPHEMERAL --reboot` → `talosctl bootstrap`. Document this in the module README until the lifecycle fix lands.
- **Talos module's worker ASG doesn't auto-refresh on LT changes.** After any worker instance type/AMI change, manually: `aws autoscaling start-instance-refresh --auto-scaling-group-name <cluster>-workers --preferences '{"MinHealthyPercentage":0,"InstanceWarmup":60}'`. Fix is a one-block add to the module.
- **`terraform output -raw` on sensitive outputs emits ANSI color codes in stderr that sometimes leak into stdout.** Use `terraform state pull | python3 -c ...` to extract sensitive fields cleanly; avoid `terraform output -raw` for any output consumed by a YAML/JSON parser.
- **SM kubeconfig secret has two representations of the endpoint.** `host` field (structured, correct after apply) vs `kubeconfig_raw.server` field (stale YAML blob generated before CP replacement finalizes). Prefer structured-field consumers; treat `kubeconfig_raw` as cache that may lag.
- **`kubectl config view --flatten` is first-occurrence-wins for name conflicts.** After merging a new cluster's kubeconfig into `~/.kube/config`, verify the cluster entry's `server:` field and use `kubectl config set-cluster` to overwrite if the old one stuck.
- **`eks.amazonaws.com/role-arn` annotation is a promise, not a contract.** Without a mutating admission webhook (pod-identity-webhook) or manual env-var + token-volume injection in the pod spec, the annotation has zero runtime effect. On a self-managed non-EKS cluster, IRSA is BYO-webhook.
