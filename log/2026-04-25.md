### S3 module + langfuse role landed in TF

New module `Infrastructure/modules/aws/s3/` mirroring `secrets-manager` shape. Creates bucket with versioning + AES256 + bucket-key + full public-access block + default lifecycle (noncurrent versions → Glacier IR after 30d, deleted after 365d). Module owns its own rw/ro IAM policies (same pattern as `secrets-manager` owning `read`/`write` policies). Outputs both policy ARNs for downstream `app_role` consumption.

Platform-level (not application-level — langfuse is a platform service for nebulosa, not portable):
- `platform/nebulosa/aws/s3.tf` invokes the new module for `cloudhaven-langfuse-blobs`.
- `platform/nebulosa/aws/iam.tf` adds `module.langfuse_role` — single role `nebulosa-langfuse`, trust scoped to `system:serviceaccount:langfuse:langfuse` (the SA the chart creates, also the SA ESO's SecretStore references — one SA, one role, two consumers).
- `policy_arns = [langfuse_secret.read_policy_arn, langfuse_blobs.read_write_policy_arn]` — both attached managed policies, no inline.

**Rule learned (composition pattern):** AWS resource modules in this repo own their own IAM policies and output the ARNs. Consumer `app_role` modules just list ARNs in `policy_arns`. No bespoke `inline_policy_json` per consumer when an upstream module can express the access. Keeps policy authoring near the resource it protects + makes the consumer dir small + free for-each across multiple roles.

### S3 module review feedback applied

User caught three issues on first version of `modules/aws/s3`:

- **`description` ≠ tag.** First draft added a `Description` tag from the variable. AWS tags should be operational metadata (Account, Cluster, ManagedBy), not free-text purpose statements. Variable kept as a Terraform-level descriptor that flows into the IAM policy `description` fields (rw/ro policies show up in console as e.g. `"Read/write access to s3://nebulosa-langfuse-blobs — Langfuse blob storage..."`), never as a bucket tag.

Also dropped a defensive `depends_on = [aws_s3_bucket_public_access_block.this]` from the bucket policy — TF derives the dependency from the shared `aws_s3_bucket.this.id` reference. Was paranoia, not necessity.

Same shape as `secrets-manager` (already had `read_policy_arn`/`write_policy_arn`). Keeps policy authoring next to the resource it protects, makes consumer dirs small, supports multi-role fan-out without duplication.

### Rules learned

- **Tailscale operator's `defaultTags` is silently constrained by the OAuth client's `tags` allow-list.** If `defaultTags` references a tag the OAuth client can't assign, behavior is undefined (silent fallback or registration failure). Always cross-check `tailscale_oauth_client.tags` against `operatorConfig.defaultTags` before relying on a custom tag.
- **The tailnet ACL is a singleton resource** — only one `tailscale_acl` per tailnet. Currently lives in `platform/humboldt/tailscale/acl.tf`; both clusters share it. Future `tag:k8s-nebulosa` style tags need to be added there (and to the OAuth client `tags` field), not in nebulosa-specific TF.
- **envoy-gateway's `install.yaml` bundles upstream gateway-api CRDs** that conflict with a separate standard-channel gateway-api install + its `safe-upgrades` policy. Filter them out via kustomize regex-targeted `$patch: delete`. One ownership: gateway-api dir owns upstream CRDs, envoy-gateway dir owns envoy-specific resources only.
- **Tailscale CGNAT IPs in Cloudflare DNS need `proxied = false`.** Cloudflare's edge can't reach `100.x.x.x` addresses. Records resolve publicly but only tailnet members can route — exactly the security posture we want for internal services.
- **macOS `mDNSResponder` caches NXDOMAIN aggressively.** After adding any new internal-zone DNS record, `dscacheutil -flushcache && killall -HUP mDNSResponder` before testing from the Mac. Otherwise stale negative responses haunt for minutes.

### Two-secret-backend pattern (worth remembering)

Got asked: "I don't see DB secrets in `nebulosa/langfuse` SecretsManager." Right — by design.

- **CNPG-managed `langfuse-pg-app` K8s Secret** holds postgres creds (11 keys: username, password, host, dbname, uri, jdbc-uri, pgpass, etc.). Auto-generated on Cluster bootstrap, lives only in the cluster, never round-trips through AWS. Langfuse's `postgresql.auth.existingSecret: langfuse-pg-app` reads it directly.
- **`nebulosa/langfuse` SecretsManager** holds *only* the 5 app-layer secrets the chart can't generate itself (salt, encryption-key, nextauth-secret, clickhouse-password, redis-password). Pulled by ESO into a K8s Secret, also named `langfuse-secrets`.
- **Two backends, two rotation cadences, zero overlap.** Putting DB creds in SecretsManager too would require a sync job between CNPG → SM → ESO → K8s Secret → Langfuse, with race conditions and stale-read windows. Avoid.

**Rule:** Operator-managed secrets stay in-cluster (CNPG, cert-manager, kubernetes.io/tls). Application-layer secrets that humans/scripts generate go to AWS Secrets Manager (or SSM for single-key cases). Cross the boundary only when an external system needs to consume — never for the internal-only credential lifecycle.

### Rules learned

- **Always pass `--context=` explicitly when working two-cluster.** kubectl context drift cost a 5-min cleanup detour today (Cluster CR landed on humboldt's `kube-system` because that's the default ns of `admin@humboldt`). Rule applies even harder when the kustomization renders resources without `metadata.namespace`.
- **Every namespaced template in a chart must have `namespace: {{ .Release.Namespace }}` explicitly.** Helm + kustomize + kubectl don't backfill it; whatever the apply context's default ns is wins. Chart auditing rule: `grep -L "namespace: {{ .Release.Namespace }}" templates/*.yaml` to spot templates missing the line, cross-reference against namespace-scoped Kinds.
- **`kustomize build --enable-helm` ≠ `helm install`.** The render-then-kubectl-apply path leaves no Helm release in storage, so `helm uninstall` doesn't work for cleanup. Use `kustomize build ... | kubectl delete -f -` (or be explicit and delete by kind/name) when you need to undo. Worth remembering before someone tries to roll back via `helm rollback` and finds nothing.
- **First chart-releaser run on a fresh repo needs `gh-pages` to exist.** API one-liner is the cheap way; orphan branch via local git is cleaner but functionally identical.
- **Bootstrap secrets that can't be committed are tech debt.** Stop accumulating them. Once one bootstrap-by-hand secret exists on a cluster, the next addition compounds runbook complexity faster than the user gets value from the feature. When the next step requires growing the kubectl-create surface, pause and wire up ESO instead.
- **Resource modules own their own IAM policies; consumer roles list the ARNs.** New `modules/aws/s3` mirrors the existing `secrets-manager` pattern — module emits an `aws_iam_policy` per access flavor (rw, ro), outputs the ARN, consumer's `app_role` lists `policy_arns = [...]`. Keeps the policy authoring next to the resource it protects, makes consumer dirs small, supports fan-out across roles trivially.
- **One IAM role per K8s SA, even when multiple in-cluster components share that SA.** The nebulosa-langfuse role is reused by both ESO (via `auth.jwt.serviceAccountRef`) AND the Langfuse pod (via IRSA env vars). Two distinct identities at the AWS-call layer, but one IRSA trust + one role + one SA-namespace pair. Splitting them into two roles would be unnecessary surface area when the SA is the same.
- **Add `default_tags` to every cluster's AWS provider before resources accumulate.** Backfilling tags later requires touching every TF apply individually. nebulosa's providers.tf had no default_tags block until this session — now matches humboldt's `ManagedBy/Account/Cluster/Environment` shape.
- **S3 bucket names are global.** "langfuse-blobs" is taken by someone, somewhere. Always cluster- or org-prefix homelab buckets (`nebulosa-langfuse-blobs`, `humboldt-temporal-blobs`).
