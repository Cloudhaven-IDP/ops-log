### Langfuse tracing working end-to-end in theo

Traces were null despite Langfuse being configured. The orchestrator was using the 2.x API (`lf.trace(...)`, `lf.trace(id=..., output=...)`) but theo-core pins Langfuse 4.5.1, which dropped that entire interface.

Langfuse 4.x is built on OpenTelemetry. The new API has no `.trace()` method. The equivalent pattern is:

- `langfuse.propagate_attributes(user_id=..., session_id=..., trace_name=...)` — module-level context manager that attaches trace metadata to the OTEL context
- `lf.start_as_current_observation(name=..., as_type="span", input=...)` — creates an observation within that context
- `lf.get_current_trace_id()` — reads the trace ID off the current OTEL span so we can store it in state
- `lf.set_current_trace_io(input=..., output=...)` — stamps the trace-level input/output before exiting
- `lf.flush()` — ships the data after the context managers close

Found these by inspecting the actual 4.5.1 package with `docker compose run --rm orchestrator python -c "import inspect; ..."` rather than guessing from docs. The key non-obvious piece was `propagate_attributes` — it's a module-level function, not a method on the client, and it's the only way to attach `user_id` and `session_id` to a trace in this version.

Also fixed two bugs that were hiding under the Langfuse error:

One: the API was sending `{"user_id": ..., "message": "...", "session_id": null}` to the orchestrator's `/run` endpoint. `ConversationState` has no `message` field — it has `messages: list[Message]`. Pydantic silently dropped the extra field, so the graph ran with an empty messages list every time. Fixed in `api/routes/chat.py` to send `messages: [{"role": "user", "content": message}]`.

Two: `server.py` referenced `state.message` in the Langfuse input dict, which crashed before the graph even ran. Fixed to `state.messages[-1].content if state.messages else ""`.

After both fixes: `POST /chat` returns a real `trace_id`, and the trace appears in the Langfuse UI under `orchestrator.run` with the correct user and input.

### build-base-images workflow yq bug

Triggering the rebuild surfaced an unrelated bug in `Infrastructure/.github/workflows/build-base-images.yml`. The matrix-compute step ran:

```
images=$(yq -o=json '[.images[].name]' Dockerfiles/images.yaml)
echo "images=$images" >> "$GITHUB_OUTPUT"
```

`yq -o=json` pretty-prints arrays by default, producing multi-line output. `$GITHUB_OUTPUT` rejects multi-line values when written as `key=value` — you'd need a heredoc form (`key<<EOF ... EOF`). Error was the literal second line of yq's output (`'  "gha-runner"'` with a 2-space indent) being interpreted as a malformed output line.

Fix: `yq -I=0 -o=json` for compact single-line JSON. After this and the runner version bump, the build succeeded and pushed the new image SHA. K8s-Bootstrap `nebulosa/action-scale-set/kustomization.yaml` `images.newTag` got updated to point at the new SHA, ArgoCD reconciled, ARC rolled the AutoscalingRunnerSet template, and the next runner pod connected to GitHub successfully.

### Makefile shell mismatch on the runner

With OIDC fixed, the workflow ran the make target — and tripped on `[[ -n "$(APP)" ]]`. The runner image is Ubuntu, where `/bin/sh` is dash. ### Kubeconfig leaked into the public workflow log

This one stung. The workflow's `aws-secretsmanager-get-secrets` step was fetching the kubeconfig with alias `KUBECONFIG`. That alias becomes an env var in subsequent steps. `KUBECONFIG` is also the env var kubectl natively reads to find its config file path. So in the next steps, kubectl saw `KUBECONFIG=<entire base64 kubeconfig YAML>` and tried to `open()` the literal YAML content as a filename. Got "file name too long" — and dumped substantial chunks of the cert/key data into stderr, which landed in the workflow log. K8s-Bootstrap is a public repo. So the cluster credentials are now in the public history.

Tailscale is the perimeter, so nobody can actually reach the apiservers from outside the tailnet, but that's not the point — the credentials are compromised in the hygiene sense and need rotation regardless.

Fixed the workflow: renamed the alias to `KUBECONFIG_BLOB` so it never collides, and the "Write kubeconfig" step now also writes `KUBECONFIG=$HOME/.kube/config` to `$GITHUB_ENV` so kubectl always sees a path. Two cluster cert rotations queued as follow-ups (k3s on nebulosa, talos on humboldt), plus re-pushing the rotated kubeconfigs to Secrets Manager.

### Make diff translates kubectl exit 1 to recipe failure

Last hurdle on the diff path: `kubectl diff` exits 1 when there are differences (which is the entire point — that's the success case for a diff). Make sees a non-zero recipe exit and itself exits 2. The workflow's wrapper logic only resets exit 1 to 0 for the `diff` target; exit 2 falls through. Fixed in the Makefile by capturing the kubectl exit and only failing the recipe for codes greater than 1: `kubectl diff -f - ...; ec=$$?; [[ $$ec -le 1 ]]`. Apply doesn't need this — apply has no soft-failure code, so the original behavior is correct.

### Repo split — decide and execute first

Before deploying anything else, split the infra layer away from backend code so each app is self-contained. Recommended shape — the **hybrid pattern**:
