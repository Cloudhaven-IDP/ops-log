## Project Rename: Soma — Powered by Theo

Consolidated architecture decisions from multiple planning sessions into a single reference doc (`soma_powered_by_theo_architecture.docx`). Key evolution from original "fitness agent" scope:

- **Multi-tenant from the start** — user_id as universal partition key across all stores
- **Temporal added** — session state caching (load S3 context once per session), durable session resumption
- **CDC feedback pipeline** — thumbs-down → DynamoDB → SQS → KEDA job → Qdrant episodic memory update (silent learning)
- **Clerk confirmed** for auth (Cognito rejected — DX worse)
- **Streamlit dropped** → Next.js landing page + `/chat` route for rate-limited Temporal Q&A session
- **Mac Mini Ollama removed** — embeddings via KubeAI on tainted RPi node, Bedrock for all LLM calls

## Architecture Decisions Clarified

- **Talos on raw EC2 with ASG** — following siderolabs.com/talos docs, not EKS
- **Envoy Gateway on BOTH clusters** — replaces nginx-ingress everywhere
- **Temporal must be same cluster as orchestrator** (RPi) — avoids cross-cluster latency on every message
- **Flannel on RPi, Cilium on Talos** — cross-cluster comms via Tailscale HTTP, not pod networking
- **Pi-3 (16GB/1TB, tainted)** — KubeAI embeddings model + potentially local Postgres (USDA nutrient DB) + Redis
- **CDC feedback = silent learning (option B)** — update Qdrant episodic memory, don't push corrected response to user
- **YAML routing table** as ConfigMap via ArgoCD — new intents = one-line PR, no code change

### Key design decisions in the code

- **Model router** (`core/model_router.py`): every LLM call goes through `get_router().invoke(task, prompt)`. `_TASK_ROUTING` dict is the authoritative mapping. Haiku for classification/parsing ($cheap), Sonnet for reasoning/vision ($moderate), Titan for embeddings.
- **Orchestrator graph** (`agents/orchestrator.py`): 4 nodes — `classify_intent → inject_memory → route_to_agent → apply_personality`. Intent classification uses Haiku at temperature=0. Personality polish only fires if Haiku detects robotic tone (cost gate). Subagent dispatch via HTTP to AWS cluster services.
- **Nutrition agent** (`agents/nutrition.py`): own FastAPI app (runs as separate Deployment). Photo flow: image → Claude Vision (Sonnet) → food description → Nutritionix API → macros → DynamoDB. Text flow: user text → Haiku parse → Nutritionix → DynamoDB.
- **Auth** (`api/auth.py`): Clerk JWT verification via PyJWKClient. Dev mode accepts `X-User-Id` header for testing without Clerk.
- **Memory** (`tools/memory.py`): Qdrant collection `soma_memory`, 768-dim Titan Embed V2 vectors, cosine similarity ≥0.60, idempotent upsert via SHA256 point IDs.
- **YAML routing table** (`routes.yaml`): intent → agent mapping, deployed as ConfigMap. Fallback chain per intent. Adding a new intent is a one-line PR.

## Talos Terraform Module — Complete

Built `modules/talos/` — reusable module for spinning up a Talos Linux cluster on AWS EC2.

Based on the [Siderolabs contrib example](https://github.com/siderolabs/contrib/tree/main/examples/terraform/aws) but adapted to project conventions:

- Uses `siderolabs/talos` provider (~> 0.7) for machine secrets, config generation, config apply, bootstrap, kubeconfig, and health check
- Individual EC2 instances (not ASG) — Talos config apply needs stable IPs for bootstrap. Control plane and workers both use `aws_instance` directly.
- NLB for Kubernetes API (443 → 6443) — NLB instead of classic ELB from the Siderolabs example
- Two security groups: intra-cluster (all traffic + Talos API port 50000) and k8s API (443)
- Official Sidero Labs AMI auto-lookup by region, overridable with `ami_id`
- Config patches support: all-nodes, CP-only, and worker-only
- Supports Tailscale-only access: `associate_public_ip = false` with Tailscale CIDR for API access
- `terraform fmt` clean, `terraform validate` passes

```
modules/talos/
├── main.tf          # SGs, NLB, EC2 instances, all Talos provider resources
├── variables.tf     # cluster_name, env, VPC, subnets, instance types, counts, CIDRs, patches
├── outputs.tf       # kubeconfig, talosconfig, endpoint, IPs, SG ID
├── versions.tf      # aws ~> 5.0, talos ~> 0.7
└── README.md        # Usage example, input/output tables
```

Design decision: placed at `modules/talos/` not `modules/aws/talos/` — Talos is its own domain concern, not a generic AWS primitive. Same pattern as `modules/twingate/` and `modules/cloudflare/`.

## Centralized Helm Chart (soma-app)

Built `gitops/charts/soma-app/` — one chart, every Soma service calls it.

```
charts/soma-app/
├── Chart.yaml
├── values.yaml                  # All defaults, everything disabled
└── templates/
    ├── _helpers.tpl             # name, labels, selectorLabels, validation
    ├── deployment.yaml          # Full: security contexts, probes, envFrom, configMap mount
    ├── service.yaml             # ClusterIP
    ├── serviceaccount.yaml      # + IRSA annotation placeholder
    ├── env-configmap.yaml       # Plain env vars as ConfigMap → envFrom
    ├── externalsecret.yaml      # ESO ExternalSecret (enabled via values)
    ├── scaledobject.yaml        # KEDA ScaledObject (enabled via values)
    ├── hpa.yaml                 # HPA autoscaling/v2 (enabled via values)
    └── configmap.yaml           # Config file mounts (e.g. routes.yaml)
```

Key design decisions:
- **KEDA + HPA mutually exclusive** — `fail` in `_helpers.tpl`, enforced at render time
- **No CPU limit by default** — CPU throttling causes latency spikes in Python/FastAPI. Set explicitly per-app if needed.
- **readOnlyRootFilesystem: false default** — Python writes to tmp. Enable per-app once confirmed safe.
- **KEDA triggers fully configurable** — CPU, memory, SQS, anything. Each app sets its own triggers in values.
- **IRSA annotation placeholder** on every ServiceAccount — ready for the IRSA module.
- **routes.yaml mounted as ConfigMap** on orchestrator — updating intent routing = Helm values PR, no code change.

|---|---|---|---|
| orchestrator | soma.nebulosa.orchestrator | nebulosa (RPi) | none — always 1 replica |
| nutrition-agent | soma.humboldt.nutrition-agent | humboldt (AWS) | CPU 70% |

Each app: `values.yaml` (base) + `values-dev.yaml` + `values-prod.yaml` + `argocd-app.yaml`.
ArgoCD destination: nebulosa → Tailscale IP `100.86.215.90:6443`, humboldt → `kubernetes.default.svc` (in-cluster).

## Talos Module — Revised

Rebuilt `modules/talos/` with user feedback:

- **Files split by concern**: `data.tf` (AMI lookup, locals, cluster endpoint logic), `security_group.tf` (cluster SG + k8s API SG + configurable additional rules), `main.tf` (EC2 via `modules/aws/ec2`, worker ASG, NLB, Talos resources, Secrets Manager)
- **Control plane via `modules/aws/ec2`** — composes existing module, `extra_security_group_ids` carries the shared cluster SG
- **Workers as ASG** (`aws_launch_template` + `aws_autoscaling_group`) — machine config passed as userdata, workers self-configure on boot. No `talos_machine_configuration_apply` needed for workers. Cluster autoscaler tags included.
- **NLB optional** (`enable_nlb = false` default) — Cloudflare Tunnel or Tailscale used instead
- **Control plane count defaults to 1** — scale to 3 when HA needed
- **Kubeconfig stored in Secrets Manager** (`<cluster>/kubeconfig`) as JSON — other states consume via `jsondecode(data.aws_secretsmanager_secret_version.kube_api.secret_string)` → `provider "kubernetes"`
- **Additional API SG rules** configurable via `additional_api_ingress_rules` variable

## humboldt Talos Cluster — Wired Up

Added Talos cluster to `platform/humboldt/aws/`:

- **`providers.tf`** — added `siderolabs/talos ~> 0.7`
- **`talos.tf`** — calls `modules/talos` with humboldt config:
  - `t3.small` control plane (1 node), `t3.medium` workers (1–5, desired 2)
  - Private subnets — nodes reachable via Tailscale subnet router (already advertising `10.0.0.0/16`)
  - `associate_public_ip = false`
  - Talos API + k8s API restricted to `100.64.0.0/10` (Tailscale CIDR)
  - NLB disabled — all access internal via Tailscale
- **`outputs.tf`** — cluster endpoint, CP IPs, worker ASG name, kubeconfig secret ARN, cluster SG ID
- `terraform validate` passes

Bootstrap flow when applied:
1. EC2 instances spin up in private subnet
2. `talos_machine_configuration_apply` reaches CP nodes via Tailscale subnet router
3. `talos_machine_bootstrap` initialises etcd on first CP node
4. Kubeconfig pulled and stored in Secrets Manager at `humboldt/kubeconfig`
5. Cluster health check validates CP is operational
