### `/api/ask` proxy contract defined

```
POST ${AGENT_URL}/ask
Content-Type: application/json
Authorization: Bearer ${AGENT_TOKEN}   # only if set

{ "messages": [{ "role": "user" | "assistant", "content": "…" }, …] }

→ 200 OK
Content-Type: text/plain; charset=utf-8
<streamed plain-text body>
```

Proxy forwards the upstream `content-type` header. 55-second `AbortSignal.timeout` on the fetch (safe inside Next.js's 60-second `maxDuration`). No `AGENT_URL` → canned stream that explains the agent isn't wired up yet; keeps the UI alive during placeholder demos and in DR scenarios where the agent is down.

**Rule:** the proxy stays a proxy. No retries, no caching (`cache-control: no-store`), no message sanitization beyond basic role/content validation, no prompt injection guarding — that's the agent service's job, not the frontend's.

### Resume — R2 decision + YAML experience model

Chose **Cloudflare R2** over S3 for the PDF host:
- Zero egress (resume is a high-download asset; S3 egress is metered, R2 is not)
- Already Cloudflare-heavy across the stack (DNS, Tunnel, Pages previously)
- Custom-domain binding on R2 is a dashboard click (`resume.cloudhaven.work`) once DNS points at the R2 bucket
- Object writes are cheap, reads are free for end-users

Experience data itself lives in `content/experience.yaml` — a structured list of `{ company, title, location, start, end, summary, bullets, stack }` entries plus optional `certifications` + `education`. Rendered on `/resume`. Updating a bullet = edit YAML → commit → CI rebuilds the image → ArgoCD syncs. Updating the PDF = upload a new file to R2, no redeploy.

**Rule:** resume content is versioned in git (YAML), the distributable artefact (PDF) is not. Changes to the page content follow the normal GitOps flow; changes to the downloadable are a single R2 `PutObject` — decoupled so either can be updated without blocking the other.

### `Dockerfile` — standalone output, three-stage, non-root

`next.config.ts` flipped to `output: "standalone"` with `outputFileTracingIncludes` pointing `/posts/**` and `/resume` at `content/`. Three-stage Dockerfile:

1. **`deps`** — `npm ci` from lockfile, `libc6-compat` for Next's sharp/swc native modules.
2. **`builder`** — full build with `NEXT_TELEMETRY_DISABLED=1`.
3. **`runner`** — copies `.next/standalone`, `.next/static`, `public/`, and `content/` (for runtime MDX reads on dev-mode rebuilds / on-demand revalidation), runs as uid 1001 `nextjs`, exposes 3000.

Base image is `node:22-alpine`. Multi-arch (amd64 + arm64) is a buildx concern in the CI workflow, not the Dockerfile — same pattern the Python services use.

Not built or pushed this pass — that lands when the ECR bootstrap (yesterday's Priority 3) is applied and `Cloudhaven-IDP/Actions/docker-build-push-ecr` can run against this repo.

**Rule:** Next.js standalone output does NOT include files under `content/` or other non-code assets by default. Explicit `outputFileTracingIncludes` + an explicit `COPY --from=builder /app/content ./content` in the runner stage are both required — skip either and MDX files 404 at runtime despite passing `next build`.

## Rules learned today

- Frontend repos own no agent logic — API routes are proxies, full stop. System prompts, bios, RAG, model SDK calls live in the agent repo.
- UI brand and deployment name are separate concerns. Brand can be editorial / playful; repo + k8s Deployment + ArgoCD app name are pragmatic and owner-anchored (`afolabi-next`).

- No `.env.example` files in app repos. Document env-var names in the README; production config syncs from AWS Parameter Store via ESO.
- For public-facing downloads (resume PDF), **R2** over S3 unless there's a specific reason. Zero egress, Cloudflare-native, custom-domain binding is a dashboard click.
- Resume content is versioned in git (YAML renders to `/resume`). The PDF artefact is uploaded to object storage and swapped without a redeploy.
- Next.js `output: "standalone"` needs both `outputFileTracingIncludes` config AND an explicit `COPY content/` in the Dockerfile runner stage. Either alone silently breaks runtime asset reads.
- Proxy routes stay proxies — no retries, no caching, no sanitization, no prompt injection logic. The agent service owns all of that.
- No descriptive comments, ever. Including JSX section comments like `{/* title row */}`.

## Cloudflare Worker — rate-limit the LLM path (decided, not built)

Before `afolabi.cloudhaven.work/ask` goes fully public, the LLM path needs rate limiting **at the Cloudflare edge**, not at the origin. Anything else hands the cost of a scraper straight to Bedrock (or to whichever provider `afolabi-desk` ends up calling).

### Shape

- **Worker** bound to the `afolabi.cloudhaven.work/api/ask*` route (not every page load — path-matched).
- **Per-IP token bucket** backed by **Cloudflare KV**: e.g. 15 requests / 15-minute sliding window. Values over threshold return `429` with a JSON `{"error":"rate_limit"}` body from the Worker, **never touching the origin**.
- **Turnstile** (optional, but leaning yes): first request from an IP gets challenged; Worker verifies the Turnstile token server-side; cache `"verified"` in KV for 24h and route verified IPs straight through. Low-friction for humans, hard for scripts.
- **Global cap** (safety belt): across all IPs combined, cap `/api/ask` at ~200 requests/hour as a hard ceiling. If the system hits it, everyone gets the same rate-limit copy. Cost control > availability on this path.
- **Logs** via Cloudflare Workers Analytics for rate-limit events and 429 counts. No origin-side rate limiter needed — Worker is the single source of truth for this.

### Why the edge, not origin middleware

- Keeps agent compute off the critical path for abuse traffic.
- KV reads are ~10ms globally — faster than any origin-side store.
- Survives origin being down — Worker can serve a static 429 even when humboldt is offline.
- Matches the existing Cloudflare-heavy posture (DNS, Tunnel, R2 for the résumé, Turnstile if we add it).
- Cloudflare Workers free plan gives 100k requests/day, Workers KV free plan gives 100k reads/day — comfortably within budget for a personal site.

### Layout

Own folder in the afolabi-next repo: `afolabi-next/edge/ratelimit/` with `wrangler.toml`, `src/index.ts`, and a GitHub Actions workflow publishing to Cloudflare on merge to `main`. Keeps frontend + edge logic colocated; deploy as separate artefacts via separate CI jobs.

### Agent-contract implication

When the Worker 429s, it should return the same plain-text rate-limit message the origin already streams, so the `/ask` UI shows consistent copy regardless of which layer denied. Worker body:

```
"Sorry — looks like you've hit the rate limit. Give it a minute and try again. ..."
```

Same string as `rateLimitMessage` in `src/app/api/ask/route.ts`. Extract to a shared constants file the Worker build picks up during bundling, or just duplicate — it's one sentence.

**Rule:** rate-limit and abuse control live at the edge for any path that costs real money (LLM calls, metered external APIs). Origin middleware is a fallback, not the primary.
