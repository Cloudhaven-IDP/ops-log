## Today's Work

### Fixed: Loki wasn't getting pod logs from either cluster

Yesterday I worked out why Loki was empty — only Kubernetes events were arriving, no pod logs. Today I confirmed it on the live clusters and fixed it.

What was happening. Alloy tails pod log files on each node and ships them to Loki. We'd told it to look in `/var/log/pods/` using a wildcard pattern. Trouble is, the part of Alloy that opens files doesn't expand wildcards. It takes the wildcard string verbatim and tries to open it as a literal filename, which obviously fails. So no pod logs, ever, on either cluster.

How I confirmed. Alloy's own logs were full of "file not found" errors with the wildcard still in the path. Listing Alloy's open files showed zero pod logs. The positions file (where Alloy tracks how far it's read into each file) was empty.

Kubernetes events were still flowing because that pipeline uses the Kubernetes API, not file reads.

The fix. Slot `local.file_match` between the path-discovery step and the file reader. Now Alloy builds the wildcard path, `local.file_match` expands it into real filenames, the reader opens those. While I was there I added a static `cluster` label (`humboldt` or `nebulosa`) so Grafana queries can filter without guessing which pod is on which cluster.

Files changed:
- `K8s-Bootstrap/humboldt/alloy/values.yaml` and `K8s-Bootstrap/nebulosa/alloy/values.yaml`: `local.file_match` plus the cluster label.
- `K8s-Bootstrap/humboldt/alloy/nebulosa-gateway-egress.yaml`: unrelated cleanup. The Tailscale operator and our config were both trying to own one field on the Service. Took it out of our file, operator owns it cleanly now. Same shape as yesterday's ArgoCD egress conflict.

Verification:
- humboldt: 18 pod log files open in Alloy, up from 0. nebulosa: 23, up from 0.
- No "file not found" lines in Alloy's logs.
- Loki's label API returns both cluster values.
- `{cluster="humboldt"}` and `{cluster="nebulosa"}` both return live pod streams.
- Confirmed in Grafana.

Lesson. In Alloy, putting a wildcard into `__path__` and piping straight to `loki.source.file` fails silently. The relabel step is happy, the file-open step blows up later. If there's a wildcard in the path, route through `local.file_match`.

### Auto-humanizer hook for ops-log writes

Set up a Claude Code PreToolUse hook on Write that intercepts any write to `ops-log/log/*.md` or `knowledge-base/**/*.md` and forces a `/humanizer` pass on the content first. Lives at `.claude/hooks/humanize-check.sh`. Exits 2 with a message when a write hits one of those paths, exits 0 silently otherwise. Configured in `.claude/settings.json` under `hooks.PreToolUse`. Pipe-tested with five payloads, all behaved.

Why. Devlog entries kept slipping into AI-flavored prose. The hook makes humanizing a forced step instead of a polite request. Side effect: the humanizer skill itself got better feedback while doing this work about what to strip (em-dash overuse, rule-of-three, bold inline-headers).

### ARC runners — central image dir, GHCR, deployed to nebulosa

Stood up GitHub Actions runners on nebulosa. Lots of small decisions and a string of chart quirks; here's the through-line.

Image strategy. New `Infrastructure/Dockerfiles/` directory holds a Dockerfile per base image. First entry is `gha-runner` — Ubuntu base plus kubectl, kustomize, helm, argocd, talosctl, tailscale CLI, jq, yq, gh, awscli, make. A single `images.yaml` declares which images get built and where they push. One workflow walks that manifest and builds everything. Future bases (node, python, go) just add a directory and a manifest entry.

Registry choice. GHCR over Docker Hub or ECR. Free for public images, native auth via `GITHUB_TOKEN` so no PAT to rotate for the build path, no per-pull rate limits like Docker Hub has, and no AWS bill for a registry that's downstream of GitHub anyway. Image landed at `ghcr.io/cloudhaven-idp/gha-runner`.

Build workflow. `Infrastructure/.github/workflows/build-base-images.yml`. Manual trigger only (`workflow_dispatch`); dropped the cron and push triggers because rebuilds should be intentional. Tags only with `${git_sha}`, no `:latest` or floating date tags, so kustomize's `images:` block can pin to an immutable SHA and `imagePullPolicy: IfNotPresent` actually behaves. First build hit `7c84f4f0a7730b0ab473ff15db65fa307489cba5`, 4 minutes cold (no cache).

Hosting choice. Runners deploy to nebulosa, not humboldt. Pods on nebulosa already reach tailnet IPs natively (per the Tailscale investigation above), so a runner doing `kubectl --context humboldt apply ...` over the tailnet just works without sidecars. RPi compute is plenty for typical CI jobs — each node has 4 cores and 8-16Gi free.

Two scale sets, one shared image. `cloudhaven-runner` (default, small) and `cloudhaven-runner-xl` (heavier CI). Workflows pick via `runs-on: cloudhaven-runner` or `runs-on: cloudhaven-runner-xl`. The Helm release name IS the runner label, so no extra wiring. Both at `minRunners: 0` so nothing sits idle on the RPi cores. Resource requests dialed down: 100m CPU / 256Mi memory for the default runner, 500m / 1Gi for XL. Limits 1 CPU / 1Gi and 2 CPU / 4Gi respectively.

K8s structure. Single namespace `arc` for the controller and both scale sets. Started with separate `arc-systems` and `arc-runners` namespaces but the isolation argument doesn't really matter for single-org infra. Combined into one ArgoCD application `nebulosa.action-scale-set` rather than two — same lifecycle, sync once. Source path `K8s-Bootstrap/nebulosa/action-scale-set/`.

Chart quirks hit (all in one sitting):
- `controllerServiceAccount` must be set explicitly in the runner scale set values. The chart does an in-cluster lookup for the controller's deployment by label; ArgoCD's helm renderer has no cluster context so the lookup returns nothing and the template fails.
- `includeCRDs: true` needed on the controller chart in the kustomization. CRDs live in the chart's `crds/` directory, which kustomize-helm-inflate skips by default. Without this the AutoscalingRunnerSet CRs fail with "no matches for kind" because the CRD never registered.
- PodSecurity. The `arc` namespace inherited the cluster's `restricted` PSA profile. The controller deployment doesn't ship strict securityContext defaults so it warned (and would have blocked under enforce). Set the namespace's PSA labels to `baseline` — generous enough for the controller, still useful as a guardrail.
- ArgoCD app destination namespace. The shared `argocd_application` module hardcoded `destination.namespace = "argocd"`. Fine for humboldt apps, broken for nebulosa-targeted ones because `argocd` namespace doesn't exist on nebulosa and `kubectl auth reconcile -n argocd` fails. Refactored the module: `namespace` variable defaults to `null`, and `coalesce(var.namespace, var.app_name)` makes destination ns default to the app name. Outliers (action-scale-set deploys to ns `arc`) override via a small `app_namespace_overrides` map.

ArgoCD app placement. The application registration belongs in `Infrastructure/platform/nebulosa/argocd/applications.tf`, not the humboldt one, even though ArgoCD itself runs only on humboldt. Both TF roots talk to the same humboldt ArgoCD instance via the argocd provider. The split is by where the workload lives, not where ArgoCD runs. I got this wrong first pass.

Live state right now. Controller running, both listeners running, both AutoscalingRunnerSets created. `cloudhaven-runner` has 1 PENDING runner in ImagePullBackOff because the GHCR package is private; resolves once the GHCR pull-secret wiring below has a real token in SSM. ExternalSecret for the GitHub App creds synced cleanly via IRSA on the nebulosa OIDC provider.

GHCR pull secret. Wrote the manifest for an ExternalSecret plus a dockerconfigjson template that pulls a PAT from SSM (`/restricted/cloudhaven/github/arc/ghcr-token`) and lands a `ghcr-pull` Secret in the `arc` namespace. Both runner pod templates reference it via `imagePullSecrets`. The PAT itself isn't in SSM yet — needs to be created manually with `read:packages` scope only. Existing IAM policy already covers `/restricted/cloudhaven/github/arc/*` so no IAM change.

Memories saved during this work:
- Never print SSM values or list `/restricted/*` paths.
- Immutable image tags (git SHA), paired with `imagePullPolicy: IfNotPresent`.
- ARC runner scale set needs `controllerServiceAccount` pinned.
- ArgoCD app placement convention (per-cluster TF root matching workload).
- Devlog in plain English (the humanizer hook enforces it now).
