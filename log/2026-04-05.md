## Tailscale on nebulosa (Phase 1 started)

Installing Tailscale on all 3 RPis rather than running a single subnet router. Reasons:
- pi-1 is the k3s control plane — if it's also the only subnet router and goes down during an upgrade, we lose all Tailscale access to the home network at the worst moment
- ArgoCD in humboldt (AWS) needs a stable address to reach the nebulosa k8s API — a direct Tailscale IP per node is more reliable than routing through a single subnet router
- SSH to any node from anywhere independently, regardless of what the other nodes are doing

Could this be Terraform? Partially. The Tailscale provider can manage auth keys, ACL policies, and autoApprovers. AWS EC2 instances get Tailscale via user_data (cloud-init) — 100% Terraform. RPis are bare metal that Terraform didn't provision, so install is manual (or Ansible). We'll use the Tailscale Terraform provider for the humboldt EC2 subnet router and policy management.
