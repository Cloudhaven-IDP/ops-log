## Tailscale on nebulosa (Phase 1 started)

Installing Tailscale on all 3 RPis rather than running a single subnet router. Reasons:
- pi-1 is the k3s control plane — if it's also the only subnet router and goes down during an upgrade, we lose all Tailscale access to the home network at the worst moment
- ArgoCD in humboldt (AWS) needs a stable address to reach the nebulosa k8s API — a direct Tailscale IP per node is more reliable than routing through a single subnet router
- SSH to any node from anywhere independently, regardless of what the other nodes are doing

Could this be Terraform? Partially. The Tailscale provider can manage auth keys, ACL policies, and autoApprovers. AWS EC2 instances get Tailscale via user_data (cloud-init) — 100% Terraform. RPis are bare metal that Terraform didn't provision, so install is manual (or Ansible). We'll use the Tailscale Terraform provider for the humboldt EC2 subnet router and policy management.

## Talos Bootstrap on pi-1 (Phase 2 started)

Decided to run Talos Linux on all 3 RPi 5s instead of reinstalling k3s — consistent tooling with the future humboldt AWS cluster, immutable OS, no drift.

Built a custom Talos image via factory.talos.dev with Tailscale, iscsi-tools (Longhorn), and util-linux-tools baked in as system extensions. Flashed pi-1, bootstrapped the control plane. pi-2 and pi-3 not yet flashed.

### Issues hit
- `ExtensionServiceConfig` is a separate Talos document type, not a field under `machine:` — caused "unknown keys" decode error until appended as its own `---` document
- DHCP gave the node a different IP at every boot — static IP config required; also hit wrong ethernet interface name (`end0` on RPi 5, not `eth0`)
- Cluster endpoint IP baked into certs at `gen config` time must match the node's actual IP — stale node entry needed manual cleanup after the static IP fix

### Result
Got pi-1 to a `Ready` Talos control plane node with correct hostname `nebulosa-pi-1` and static IP — but it was a false victory. See below.

### Late-session blocker: Talos was running entirely from SD card

When we pulled the SD card to reuse it for pi-2, everything died. Root cause, confirmed by dmesg:

```
volume "STATE", "phase": "ready -> closed", "location": "/dev/mmcblk0p5"
```

`mmcblk0` is the SD card. Talos STATE (config, certs, etcd data) was never on NVMe — it was on the SD card the whole time. The install to `/dev/nvme0n1` never actually ran.

**Why**: the NVMe still had the old Ubuntu install from the k3s wipe. Boot order is NVMe-first (`0xf146`). On first boot with the Talos SD card, the RPi tried NVMe → found Ubuntu → Ubuntu booted. Talos never got a chance to run as the installer.

Confirmed: after re-flashing the SD card and rebooting, pi-1 came straight up in Ubuntu (NVMe boot).

### Fix for next session

Pi-1 is currently running Ubuntu. SSH in and wipe the NVMe first, then boot the Talos SD card into maintenance mode and reinstall cleanly:

```bash
# SSH into pi-1 (Ubuntu, currently at 10.0.0.179)
ssh ubuntu@10.0.0.179
sudo wipefs -a /dev/nvme0n1   # clear Ubuntu from NVMe
sudo reboot                    # RPi falls through to SD card → Talos maintenance mode
```

Then apply config as before — this time the install to NVMe will succeed on a blank disk.
