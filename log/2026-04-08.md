# 2026-04-08

## Architecture Decisions

### nebulosa: k3s confirmed (Talos dropped)

Talos on bare-metal RPi hit boot order friction twice (NVMe/SD card conflict, static IP quirks).
k3s on Ubuntu is the right call for nebulosa — simpler to operate, all 3 nodes already running
Ubuntu with Tailscale installed. Talos stays for humboldt (AWS EC2) where it actually makes sense.

### Gateway API: Envoy Gateway

- CNI-agnostic — works identically on k3s (nebulosa) and Talos (humboldt)
- Strong arm64 support
- CNCF-backed, clean Gateway API spec compliance
- Consistent across both clusters — no divergence

Rejected:
- nginx Gateway Fabric — separate project from nginx-ingress, not a migration, F5/nginx Inc. direction murky
- Traefik — already skipped in favour of nginx-ingress, Gateway API support less complete
- Cilium — powerful but requires CNI swap mid-cluster; revisit if starting fresh on humboldt
- Istio — too heavy for 3 RPi nodes

### Service Mesh: Linkerd

- Lightweight, low resource overhead — important on RPi hardware
- Excellent arm64 support
- Automatic mTLS between services
- Simple to operate vs Istio
- Pairs cleanly with Envoy Gateway (Envoy owns the edge, Linkerd owns service-to-service interior)

### Public Ingress: Cloudflare Tunnel

Tailscale Funnel considered but doesn't support custom domains natively — TLS cert only covers
*.ts.net, CNAME to custom domain breaks on cert mismatch.

Cloudflare Tunnel chosen for public-facing services (e.g. about-me / Streamlit app):
- Native custom domain support (afolabi.cloudhaven.work)
- Handles TLS automatically
- Free
- Tailscale stays for all internal/admin access — two tools, two jobs

### Installation Order (when ready)

1. k3s install on all 3 nebulosa nodes (`--disable=traefik`)
2. MetalLB
3. Linkerd (control plane first, then namespace annotation for proxy injection)
4. Envoy Gateway (ensure Linkerd proxy injection enabled on Envoy namespace for mTLS on first hop)
5. cert-manager
6. Cloudflare Tunnel (for public-facing services only)

## Tailscale Terraform Apply

Applied `platform/humboldt/tailscale/` state. ACL created successfully on first apply.

Hit a chicken-and-egg error — `tailscale_device_tags` and `tailscale_tailnet_key` ran in parallel with `tailscale_acl`, but tags aren't valid until the ACL exists. Fixed with `depends_on = [tailscale_acl.this]` on both resources.

Final state (6 resources):
- `tailscale_acl` — tag owners, ACL rules, autoApprovers for `10.0.0.0/16`
- `tailscale_tailnet_key.subnet_router` — auth key for humboldt EC2, `tag:subnet-router`
- `tailscale_device_tags.nebulosa` — pi-1 → `tag:k8s-cp`, pi-2/pi-3 → `tag:k8s-worker`
- `aws_ssm_parameter.tailscale_auth_key` — key written to `/restricted/tailscale/auth-key`

## humboldt AWS Apply

Applied `platform/humboldt/aws/` state. 6 resources created:
