## Today's Work — Debugging "three IRSA-dependent workloads broken after rebuild"

User report: Tailscale operator, ESO, EBS CSI all non-functional on humboldt. Symptom framed as "the ESO role isn't set up." Actual story turned out to be three independent problems that happened to present simultaneously.

### Initial hypothesis chain (most of which was wrong)

1. **Opus's write-up in 04-22 log:** claimed the fix was installing `amazon-eks-pod-identity-webhook` as the "standard answer for non-EKS IRSA," and fabricated a "historical claim of 'we never needed this' was a latent bug" justification. User called BS correctly — they had IRSA working on nebulosa with ESO for months (documented in `blog/content/posts/building-cloud haven-05.md`, dated 2025-12-08), no webhook ever installed.
2. **My first pivot:** suggested the OIDC thumbprint was stale. Ran diagnostics, compared AWS-registered thumbprint `932bed339...` against the current leaf cert fingerprint `197d389...`, declared a mismatch, recommended `terraform apply -replace=aws_iam_openid_connect_provider.humboldt`. This was wrong — I was comparing the wrong cert.
3. **TF replace did reveal a real drift though:** stored URL was `oidc-humboldt.cloudhaven.work` (no scheme) vs config's `https://oidc-humboldt.cloudhaven.work`. Create-only attribute, so TF replaced the provider. New ARN = same ARN (OIDC provider ARNs are deterministic from hostname).

### The actual three problems

**Problem 1 — Tailscale operator starved of its OAuth Secret.**
`kubectl get secretstore,externalsecret -A` returned `No resources found`. The `K8s-Bootstrap/humboldt/tailscale-operator/` kustomization declares `secretstore.yaml` + `externalsecret.yaml`, but only the namespace had been applied post-rebuild. Tailscale operator was waiting for `operator-oauth` Secret that no one was creating. Fix: `kubectl apply -k` on the tailscale-operator kustomization. Operator came up cleanly within seconds — which **retroactively proved the OIDC + IAM + ESO-native-JWT chain was fine all along**.

**Problem 2 — ESO wasn't broken at all.**
ESO pod had no `AWS_ROLE_ARN` / `AWS_WEB_IDENTITY_TOKEN_FILE` env vars, no `sts.amazonaws.com`-audience projected token. Opus had flagged the absence of env vars as a problem. **It isn't, for ESO.** ESO's `auth.jwt.serviceAccountRef` mode does not use the AWS SDK credential chain — it calls the K8s `TokenRequest` API itself for the referenced SA, then does `AssumeRoleWithWebIdentity` directly. No env-var injection needed on ESO's own pod. This is what made IRSA work on nebulosa without a webhook for the last 5 months.

**Problem 3 — EBS CSI was genuinely broken (and probably never really worked).**
Live pod logs: `no EC2 IMDS role found`. Pod spec had no IRSA env vars, no projected token. AWS SDK Go credential chain: empty static env → no web identity env → IMDS 404 (no instance profile on worker, confirmed by `grep iam_instance_profile modules/talos/ platform/humboldt/` = empty) → fail. No `amazon-eks-pod-identity-webhook` on the cluster, no mutating admission webhook of any kind. The `eks.amazonaws.com/role-arn` annotation on the SA was decorative — nothing was reading it to inject credentials. User's memory of "EBS CSI worked before the rebuild" almost certainly means "the controller was Running and we never requested a PVC, so the broken credential path was never exercised." Chart version 2.58.0's health check calls `DescribeAvailabilityZones` at startup, which is what finally surfaced the failure.

### Fix for EBS CSI — chart-level env + projected token

Patched `K8s-Bootstrap/humboldt/ebs-csi-driver/values.yaml`:

```yaml
controller:
  serviceAccount:
    annotations:
      eks.amazonaws.com/role-arn: "arn:aws:iam::445746982355:role/humboldt-ebs-csi-controller"
  env:
    - name: AWS_ROLE_ARN
      value: "arn:aws:iam::445746982355:role/humboldt-ebs-csi-controller"
    - name: AWS_WEB_IDENTITY_TOKEN_FILE
      value: /var/run/secrets/eks.amazonaws.com/serviceaccount/token
  volumes:
    - name: aws-iam-token
      projected:
        sources:
          - serviceAccountToken:
              audience: sts.amazonaws.com
              expirationSeconds: 3600
              path: token
  volumeMounts:
    - name: aws-iam-token
      mountPath: /var/run/secrets/eks.amazonaws.com/serviceaccount
      readOnly: true
```

`kubectl kustomize --enable-helm | kubectl apply -f -` + rollout restart → 5/5 Running, no CrashLoop.

### Rules learned

- **ESO native JWT mode vs AWS SDK credential chain are two fundamentally different IRSA paths on non-EKS clusters.** ESO (and anything that implements the JWT-to-STS dance itself) does not need env-var injection. Workloads using stock AWS SDK Go / boto3 *do*. Assuming one universal fix for both is the core mistake Opus made yesterday. The right mental model: "does this pod call AWS SDK directly, or does it read a K8s Secret that ESO populated?" Only the first category needs env vars + projected token.
- **On non-EKS clusters you have three ways to feed AWS SDK pods IRSA creds — pick per-use-case, not universally.** (1) `amazon-eks-pod-identity-webhook` as a mutating admission webhook — worth it at ~3+ SDK-using workloads. (2) Explicit `env` + `volumes` + `volumeMounts` in the Helm values for each chart — best when it's 1–2 workloads. (3) Node instance profile — broadest blast radius, fine for throwaway clusters, wrong for production-shaped homelabs. Today EBS CSI is the only SDK-using workload on humboldt, so option 2. Revisit option 1 when theo-agents / nutrition / coach land.
- **Opus's "non-EKS clusters need the webhook" framing was lazy pattern-matching.** The correct nuance: *SDK-credential-chain* pods need the webhook (or an equivalent); JWT-native pods (ESO, anything that calls `sts:AssumeRoleWithWebIdentity` itself) don't. The nebulosa blog post would have surfaced this distinction to Opus if it had actually read the user's prior work instead of rationalizing with a fabricated backstory.
- **TF `tls_certificate` data source returns the chain *root-first*.** `certificates[0]` is the GTS Root R4 cert, not the leaf — AWS OIDC provider thumbprint registration deliberately targets a cert stable across leaf rotations. Don't compare `openssl s_client | openssl x509 -fingerprint` output (leaf) to the AWS-registered thumbprint (root) and declare a mismatch. To verify thumbprint correctness, dump the full chain with `openssl s_client -showcerts` and check every cert's SHA-1 against AWS's registered value.
- **AWS provider upgrade normalized OIDC provider URLs.** Previously stored state had `oidc-humboldt.cloudhaven.work` (no scheme); config now wants `https://...`. URL is a force-new attribute, so a harmless drift shows up as a "replace" diff. Same final ARN, so downstream IRSA roles are unaffected. Let it replace, no harm done.
- **After any bootstrap re-apply, `kubectl get clustersecretstore,secretstore,externalsecret -A` is a smoke test worth running.** If those return `No resources found` but your kustomizations declare them, the manifests never actually reached the API server — even if the namespace shows Active. Opus's triage table on 04-22 line 568 ("tailscale-operator ✅ namespace Active") was misleading for this reason: "namespace Active" ≠ "resources applied."
- **EBS CSI driver v2.58.0 crash-loops visibly when it can't get creds, older versions didn't.** Version 2.58.0 added a `DescribeAvailabilityZones` call in the readiness probe path. If PVC provisioning was never actually exercised on the old cluster, the broken IRSA was asymptomatic. Rebuilds are a good forcing function for these latent issues.
- **Don't `terraform apply` with IAM role modules commented out.** The `humboldt-tailscale-operator` and `humboldt-ebs-csi-controller` roles are live in AWS; commenting the modules and applying would destroy them mid-debug. If you comment modules to work around a target-apply dependency error, uncomment them immediately after the targeted step.

## humboldt CP bump — `t4g.small` → `t4g.medium`

CNPG operator was crash-looping on humboldt with `Failed to renew lease, context deadline exceeded`. Dug into apiserver logs, found:

```
grpc: addrConn.createTransport failed to connect to 127.0.0.1:2379
transport: authentication handshake failed: context canceled
```

apiserver couldn't reliably reach etcd. Controller-manager + scheduler each restarting 7x in 20 seconds. Root cause: 2 GiB RAM on `t4g.small` CP is too tight for etcd + kube-apiserver + controller-manager + scheduler + Talos runtime, exactly as flagged earlier in this log's control-plane-memory-watch section.

**Surprise — same-family resize is in-place, NOT a destroy/replace.** The 04-22 log's rule *"any CP instance type change wipes etcd"* turned out to be specific to the x86→ARM arch change (forced AMI swap). Within the `t4g` family, TF showed `~ update in-place`, AWS does stop/start, etcd is preserved. Total downtime ~1m14s TF modify + ~3-5 min for CP to fully rejoin. No bootstrap re-apply, no K8s-Bootstrap sweep. Cost delta: ~$12/mo.

Post-bump state: apiserver / CM / scheduler / CNPG all Running, no crash loops. Node capacity updated from 1.8 GiB → 3.72 GiB (kubelet needed a beat to re-register).

**Rule correction to the 04-22 log:** "CP instance type change = destroy/replace" is **only** true when the AMI would also change (arch swap, OS family swap). Same-arch same-AMI resize is in-place via stop/start. `instance_type` is not a force-new attribute in the AWS provider — the rebuild runbook should distinguish these two cases.

## Decision to park — chart publishing design

Planned to `helm install` the postgres-platform chart directly into nebulosa, user course-corrected: **everything in K8s-Bootstrap should be declarative, and the chart should be consumed via a resolvable Helm repo URL, not vendored or installed imperatively.**

Decided on **GitHub Pages as the Helm repo host for GitOps/charts/\***. Pattern:
1. GitHub Actions workflow in GitOps repo runs on merge — `helm package`, `helm repo index`, push to `gh-pages` branch.
2. K8s-Bootstrap kustomizations reference `repo: https://cloudhaven-idp.github.io/GitOps` in their `helmCharts:` blocks (same shape as how we consume jetstack, cnpg, langfuse upstream charts today).
3. `postgres-platform` becomes the first in-house chart consumed this way; future charts (tenant variants, app wrappers) follow.

Rejected alternatives: GHCR OCI (user-facing private-by-default overhead not worth it for homelab), vendoring charts into K8s-Bootstrap (duplicates content, diverges).

**This is tomorrow's starting point.** No code landed for the publish workflow or the K8s-Bootstrap/nebulosa/postgres-langfuse dir yet — only the chart patches in GitOps.
