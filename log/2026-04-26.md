### CoreDNS ‚Äî architecture decision

**Problem:** in-cluster pods resolving `*.internal.cloudhaven.work` hairpin out through Tailscale (`100.85.33.92`) and back. Need CoreDNS to resolve these hostnames to the Envoy Gateway's ClusterIP internally.

**ExternalDNS considered and rejected (for now).** ExternalDNS watches HTTPRoutes/Services and auto-creates Cloudflare records from the Gateway's LB IP ‚Äî good DRY property (hostname declared once in HTTPRoute, not also in Terraform). But it would conflict with Terraform owning the same records. Pick one owner. Given all `*.internal.cloudhaven.work` records point to the same single IP and the `for_each` Terraform approach is already clean, ExternalDNS would add operational surface area without meaningful benefit at current scale. Revisit if the number of services grows large enough that the Terraform list becomes annoying.

**Controller/operator considered and rejected.** First instinct was a long-running Deployment watching HTTPRoute events and reconciling CoreDNS on change. Overkill ‚Äî requires a reconcile loop, watch API handling, leader election for reliability. More code than the problem warrants.

**Decision: `coredns-syncer` ‚Äî tiny HTTP server, curl-driven.** A ~50-line Python script exposing `POST /add` and `POST /remove`. Uses the same kubectl + awk + marker pattern already proven in the `postgres-platform` chart's coredns-rewrite-job. No background polling, no watch loop ‚Äî you call it when you add a service, it patches the CoreDNS ConfigMap, done.

```bash
curl -X POST http://coredns-syncer:8080/add \
  -d '{"hostname": "grafana.internal.cloudhaven.work"}'
```

Chart is idempotent and generic ‚Äî future one-off kubectl-patching jobs can reuse the same shape by passing different tasks. Chart name: `coredns-syncer`. Script code lives in the scripts repo ‚Üí built ‚Üí pushed to ECR ‚Üí K8s pulls it.

Still need: Envoy Gateway Service name in `envoy-gateway-system` (`kubectl get svc -n envoy-gateway-system`) to wire the rewrite target FQDN.

### CoreDNS automation ‚Äî parked, manual for now

Explored full automation: controller watching HTTPRoutes ‚Üí patches CoreDNS ConfigMap, packaged as a generic `nucleus-jobs` Helm chart (CronJob or dispatcher + Job pattern). Decided it's overkill for a task that runs a handful of times total. Deleted the half-written chart.

**Manual workflow instead** ‚Äî one-liner using the same kubectl + awk + marker pattern from the postgres-platform coredns-rewrite-job:

```bash
ALIAS="<hostname>.internal.cloudhaven.work"
TARGET="<envoy-svc>.<namespace>.svc.cluster.local"  # kubectl get svc -n envoy-gateway-system
MARKER="# coredns-syncer"
CORE=$(kubectl -n kube-system get cm coredns -o jsonpath='{.data.Corefile}')
LINE="    rewrite name exact ${ALIAS} ${TARGET} ${MARKER}"
PATCHED=$(printf '%s' "$CORE" | awk -v L="$LINE" '!done && /^\.:53 \{/ { print; print L; done=1; next } { print }')
kubectl -n kube-system create cm coredns --from-literal=Corefile="$PATCHED" --dry-run=client -o yaml | kubectl -n kube-system apply -f -
```

CoreDNS `reload` plugin picks it up within ~30s. Re-run with a different `ALIAS` for each new service.

The `nucleus-jobs` automation pattern (generic base chart ‚Üí CronJob or dispatcher + Job, image in ECR) is the right long-term shape and good wiki material. Parking until there's a second or third job that justifies the overhead.

**Cross-cluster egress (humboldt ‚Üí nebulosa) ‚Äî architecture note:**

Talos EC2 nodes have no native Tailscale ‚Äî no `tailscale0` interface, no route for `100.64.0.0/10` in the kernel. Pods on humboldt cannot reach Tailscale IPs directly even though the ACL (worker‚Üíworker) and subnet router are in place. Contrast with the RPi nodes (nebulosa): each Pi runs native Tailscale, so pods on nebulosa can reach any tailnet device without proxies.

The Tailscale operator's egress proxy pattern is the Talos workaround: a pod that IS a tailnet device forwards traffic from the cluster network into the tailnet. One proxy per target IP is sufficient ‚Äî all HTTP services behind the same gateway share the proxy, differentiated by CoreDNS rewrite per hostname.

**`humboldt/alloy/nebulosa-gateway-egress.yaml`** ‚Äî `Service` annotated `tailscale.com/tailnet-ip: "100.85.33.92"` (nebulosa-internal). Operator created proxy pod `ts-nebulosa-gateway-4vqnc-0` in the `tailscale` namespace. CoreDNS rewrite added on humboldt: `loki.internal.cloudhaven.work` ‚Üí `nebulosa-gateway.kube-system.svc.cluster.local`. Alloy writes now clean.

Any future humboldt service needing to reach a nebulosa HTTP service (Grafana, Langfuse, etc.) reuses the same `nebulosa-gateway` proxy ‚Äî just add another CoreDNS rewrite for that hostname.

**Twingate comparison note (for knowledge-base):** This egress proxy boilerplate is a known Tailscale-on-k8s friction point. Twingate handles this differently ‚Äî each internal service is a named Twingate Resource accessible to any authenticated connector without per-service proxy plumbing. Worth capturing in the architecture decisions wing.

**Decision: Loki on S3, not local-path.** Chunks/index survive node loss and pod rescheduling. WAL PVC on local-path is acceptable (ephemeral state, Loki replays from S3 on loss).

**Decision: Grafana credentials in SSM Parameter Store, not Secrets Manager.** Consistent with qdrant and tailscale operator patterns (secrets ‚Üí `/restricted/`).

### Tailscale device model ‚Äî clarified

Individual Pi devices (`pi-1/2/3`) and the operator-created `nebulosa-internal` device serve different layers:

### Pod log file tailing ‚Äî broken on both clusters (investigation 2026-04-27)

**Symptom:** `{namespace="argocd"}` in Grafana returns only kube events (80 entries, `job=integrations/kubernetes/eventhandler`). Zero pod log lines. Same on nebulosa ‚Äî all namespace hits in Loki are from kube events, not file tailing.

**Confirmed:**
- CoreDNS rewrite for `loki.internal.cloudhaven.work` in place on humboldt ‚úì
- Alloy pods running 2/2 on both clusters ‚úì
- `/var/log/pods` is mounted and log files exist (e.g. `argocd_argocd-application-controller-0_.../application-controller/0.log`, `-rw-r-----`) ‚úì
- Shell glob `ls /var/log/pods/*ea89e72c*/application-controller/*.log` resolves correctly ‚úì
- File is readable (`cat` works) ‚úì
- Zero open file descriptors to `/var/log/pods/**` in the Alloy process (`ls /proc/1/fd | grep var/log` ‚Üí empty) ‚úì ‚Äî confirms no files are being tailed

**Root cause:** `loki.source.file` calls `os.Stat()` on the raw `__path__` value before glob expansion. The current path template is `/var/log/pods/*$1*/$2/*.log` ‚Äî contains `*` wildcards. `os.Stat()` on a literal glob string always returns ENOENT. Alloy logs `"failed to tail file, stat failed"` and skips the target without ever attempting glob expansion. The positions file at `/tmp/positions.yaml` is empty on both clusters (nothing ever tailed).

Kube events work because `loki.source.kubernetes_events` uses the k8s API, not file I/O.

**Fix (pending validation):** Replace the glob-based path rule with an exact path rule using all four identifying labels. Containerd path is deterministic: `namespace_podname_uid/container/0.log`:

```river
rule {
  source_labels = ["__meta_kubernetes_namespace", "__meta_kubernetes_pod_name", "__meta_kubernetes_pod_uid", "__meta_kubernetes_pod_container_name"]
  separator     = "/"
  regex         = "([^/]+)/([^/]+)/([^/]+)/([^/]+)"
  target_label  = "__path__"
  replacement   = "/var/log/pods/$1_$2_$3/$4/0.log"
}
```

Also add `cluster` label (static, per-cluster) so humboldt and nebulosa logs are distinguishable in Grafana. Apply to both `humboldt/alloy/values.yaml` and `nebulosa/alloy/values.yaml`.
